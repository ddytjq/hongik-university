<p> Now assume that <code>/ABC</code> is the current directory. When we try to 
  compile <code>Tester.java</code> with the following command,</p>
<blockquote>
	
  <pre>javac Tester.java</pre>
</blockquote>
<p>the compiler displays this error message:</p>
<blockquote>
	
  <pre>Tester.java:7: var is not public in def.A; cannot be accessed from outside package
      a.var = 1;
       ^
1 error</pre>
</blockquote>
<p>Explain why the error message occurs and how to eliminate it.
<span style="font-style:italic"></span>

<p class="answer">Answer 1: Package Visibility</p>
<div class="postSubtitle">The error occurs because the variable, <code>var</code>, in the <code>A</code> 
class has <em>package visibility</em>. A member (variable or method) with <em>package 
visibility</em> is accessible <em>only</em> to classes defined within the same 
package. In the code above, class <code>A</code> resides in the package named 
<code>def</code>. The class <code>Tester</code>, on the other hand, resides in 
what's known as the <em>default</em> package. Java considers any class missing 
a <code>package</code> statement as being in the <i>default</i> package. Because 
the default package is not the same as the <code>def</code> package, the member 
<code>var</code> in class <code>A</code> is not accessible within the <code>Tester</code> 
class.</div>
<p> The table below should give you an idea of how member visibility declarations 
  (i.e., <code>public</code>, <code>protected</code>, package, and <code>private</code>) 
  affect where a member can be referenced. Of course, package visibility is what 
  you get if none of the other three visibility modifiers are specified. 
<p>
<div align="center">
   <table border=2 cellpadding=2>
		<tr bgcolor="#CCFFCC">
			<td rowspan=2><strong>If a member is<br>
				declared as...</strong></td><td colspan=4 align="center" bgcolor="#FFFFFF"><strong>then is the member accessible within...</strong></td></tr>
		<tr bgcolor="#FFFFFF">
			<td>its own class?</td><td>any class in its<br>
				own package?</td><td>any subclass in a<br>
				different package?</td><td>any non-subclass in<br>
				a different package?</td></tr>
		<tr>
			<td bgcolor="#CCFFCC"><code>public</code></td><td align="CENTER">YES</td><td align="center">YES</td><td align="center">YES</td><td align="CENTER">YES</td></tr>
		<tr>
			<td bgcolor="#CCFFCC"><code>protected</code></td><td align="center">YES</td><td align="center">YES</td><td style="color:red" align="CENTER">YES*</td><td align="CENTER">NO</td></tr>
		<tr>
			<td bgcolor="#CCFFCC">package</td><td align="center">YES</td><td align="center">YES</td><td align="CENTER">NO</td><td align="CENTER">NO</td></tr>
		<tr>
			<td bgcolor="#CCFFCC"><code>private</code></td><td align="center">YES</td><td align="center">NO</td><td align="CENTER">NO</td><td align="CENTER">NO</td></tr>
   </table></div>
<br>
<div style="color:red; margin-left:1.0in; margin-right:1.0in"> * When a class, 
  <em>S</em>, extends another class, <em>B</em>, every instance of <em>S</em> 
  contains <u>all</u> the members declared in <em>B</em>, regardless of the visibility 
  modifiers (e.g., <code>private</code>, <code>protected</code>, package, or <code>public</code>) 
  on those members. Even though all instances of S contain all the members declared 
  in <em>B</em>, they cannot necessarily refer to all of them. <br>
  <br>
  <img src="images/visibility-of-superclass-protected-members.gif" width="166" height="145" align="left" hspace="10" vspace="10">Let's 
  take a look at Figure 1. This figure shows four classes: <code>B</code>, <code>S</code>, 
  <code>T</code> and <code>R</code> with class <code>B</code> containing a protected 
  member, &quot;<code>protAttrib</code>&quot;. We'll use lower-case letters to 
  represent instances of the corresponding classes, proper. For example, assume 
  object <code>b</code> was created by calling <code>new B(),</code> and so on.<br>
  <br>
  An object can only refer to <code>protected</code> members in objects that are 
  of its own class or subclass. So, for example, object <code>s</code> can refer 
  to the <code>protAttrib</code> variable in itself or in object <code>r</code> 
  or in any other instances of <code>S</code> or <code>R</code> but not in any 
  instances of <code>T</code> or <code>B</code>, proper. Object <code>b</code> 
  can refer to the <code>protAttrib</code> variable in <code>b</code>, <code>s</code>, 
  <code>t</code>, <code>r</code> as well as any other instances of <code>B</code>, 
  <code>S</code>, <code>T</code>, and <code>R</code>. Object <code>r</code> can 
  only refer to the <code>protAttrib</code> in itself or other instances of <code>R</code>, 
  proper. Admittedly, this may be a bit hard to understand without an example. 
  See <a href="#question_2">Question 2</a> for a concrete example using Java code. 
</div>

Now assume that /abc is the current directory. When we try to compile <code>B.java</code> with the following command,<br>

<p class="answer">Answer 2: Member Visibility</p>
<div class="postSubtitle">When one class extends another class, instances of the subclass will contain their 
own copies of <em>all</em> members declared in the superclass (including any <code>private</code> 
variables). In the example above, all instances of <code>B</code> will contain 
all four of the variables declared in <code>A</code>. However, not all of these 
variables will be accessible (i.e., visible) by instances of <code>B</code>.</div>
<p>In the descriptions below, we say a variable is <em>inherited</em> by a subclass if it is accessible to (in other words, it can be referred to) instances of the subclass.</p>
<u>a.privateVar = 1; <span class="comment">// Line 10</span></u>
<div style="margin-left:0.4in"> <code>private</code> members are only accessible 
  from within the class in which they are declared. Because <code>privateVar</code> 
  was declared in class <code>A</code>, it is not accessible from within instances 
  of class <code>B</code>. The following compiler error results: </div>
<blockquote>
  <pre>
B.java:10: privateVar has private access in def.A
      a.privateVar = 1;    // Line 10
       ^
</pre>
</blockquote>
<u>a.packageVar = 2; <span class="comment">// Line 11</span></u> 
<div style="margin-left:0.4in"> <code>package</code> members are only accessible 
  to classes within the same package as the class that declares them. In this 
  case, the member <code>packageVar</code>, is declared in class <code>A</code>, 
  in the <code>def</code> package. <code>packageVar</code> is accessible <i><b>only</b></i> 
  from within classes in the <code>def</code> package. Since <code>B</code> is 
  in the "default package" (i.e., it has no <code>package</code> statement) it 
  does not have access to <code>packageVar</code>. The following compiler error 
  results: </div>
<blockquote>
  <pre>
B.java:11: packageVar is not public in def.A; cannot be accessed from outside package
      a.packageVar = 2;    // Line 11
       ^
</pre>
</blockquote>
<u>a.protectedVar = 3; <span class="comment">// Line 12</span></u> 
<div style="margin-left:0.4in"> Even though class <code>B</code> extends class 
  <code>A</code>, instances of <code>B</code> are not allowed access to protected 
  members in instances of <code>A</code>, proper. Because field <code>a</code> 
  is an instance of <code>A</code>, proper, a <code>instances of B</code> do not 
  have access to its <code>protected</code> members. Instances of class <code>B</code> 
  can only access <code>protected</code> members in instances of classes that 
  are of type <code>B</code>. This includes classes that inherit from <code>B</code> 
  (such as the <code>C</code> class). The following compiler error results: </div>
<blockquote>
  <pre>
B.java:12: protectedVar has protected access in def.A
      a.protectedVar = 3;  // Line 12
       ^</pre>
</blockquote>
<u>a.publicVar = 4; <span class="comment">// Line 13</span></u> 
<div style="margin-left:0.4in">
  <p><code>public</code> members are available "everywhere". Consequently, <code>B</code>'s 
    reference to <code>a.publicVar</code> compiles without error.</p>
</div>
<u>privateVar = 5; <span class="comment">// Line 17</span></u>
<div style="margin-left:0.4in"> <code>private</code> members are only accessible 
  from within the class in which they are declared. Because <code>privateVar</code> 
  was declared in class <code>A</code>, it is not accessible from within instances 
  of class <code>B</code>. Note that on Line 10 we attempted to refer to a <code>private</code> 
  member in another object (object <code>a</code>). On line 17 we are attempting 
  to refer to the copy of <code>privateVar</code> that <code>B</code> contains 
  by virtue of being a subclass of <code>A</code>. In terms of the visibility 
  of <code>privateVar</code>, however, it does not matter. <code>privateVar</code> 
  is a <code>private</code> member declared in the <code>A</code> class and can 
  <em>only</em> be accessed from within that class. The following compiler error 
  results: </div>
<blockquote>
  <pre>
B.java:17: privateVar has private access in def.A
      privateVar = 5;      // Line 17
      ^</pre>
</blockquote>
<u>packageVar = 6; <span class="comment">// Line 18</span></u> 
<div style="margin-left:0.4in"> As was true for line 11, package members are only 
  accessible to classes within the same package as the class that declares them. 
  Here an instance of <code>B</code> is trying to access the <code>packageVar</code> 
  variable it contains by virtue of being a subclass of <code>A</code>. The <code>protectedVar</code> 
  variable, however, was <em>declared</em> by a class in the <code>def</code> 
  package. Recall that <code>B</code> is not in the <i>default</i> package. Therefore, 
  <code>B</code> cannot access the <code>packageVar</code> member. The following 
  compiler error results:</div>
<blockquote>
  <pre>B.java:18: packageVar is not public in def.A; cannot be accessed from outside package
      packageVar = 6;      // Line 18
      ^</pre>
</blockquote>
<a name="line_19_description"></a> <u>protectedVar = 7; <span class="comment">// 
Line 19</span></u> 
<div style="margin-left:0.4in"> There is no compiler error for line 19. Remember 
  that a <code>protected</code> member declared in a class, <em>R</em>, is accessible 
  within: 
  <ul style="list-style-type:square">
    <li style="line-height:0.15in">any instance of any class in the same package 
      as <em>R</em> (this is exactly the same restriction that applies to package 
      members)</li>
	 <li style="line-height:0.15in">an instance, <em>m</em>, of any class, <em>M</em>, 
      such that a reference of type <i>M</i> is assignable to a reference of type 
      <i>R</i>.
  </ul>
  The language here is getting a bit abstract. In our coded example above, class 
  <em>R</em> is represented by class <code>A</code> and class <em>M</em> is represented

<p>The <strong>first statement</strong> to be executed in any Java constructor will <i><b>always</b></i>

<ol>
  <ol>

	   <li>If neither of the above two cases applies the compiler inserts an implicit 
    call to the superclass no-arg constructor as the first statement: <code>super();</code></li>
  </ol>
</ol>
From the above three cases it should be clear that if some class <em>B</em> extends 
another class, <em>A</em>, calling <em>B</em>'s constructor will result in one 
of <em>A</em>'s constructors being executed <em>in its entirety</em> before the 
second statement is <em>B</em> is ever executed.<br>
<br>

defined by the class are executed, in the order in which they are defined. After 
all instance initializers have executed the constructor body finally executes.<br>

<p>What's happening here is actually pretty straight forward. <code>A</code>'s 
  static initializers begin executing at line 2. At line 3 a new instance of <code>B</code> 
  is created. Before this instance can be created, however, <code>B</code>'s static 
  initializers need to run. The first line of output is produced by <code>B</code>'s 
  static initializer at line 16. So far, nothing here should be too surprising.</p>
<p>At line 17 a new instance of <code>A</code> is being created. But how can <code>A</code> 
  be instantiated when <code>A</code>'s static initializers have not completed? 
  Remember, the reason line 17 is being executed in the first place is because 
  <code>A</code>'s static initializer (line 3) is trying to create an instance 
  <code>B</code>. It turns out that Java <em>will</em> allow an instance of <code>A</code> 
  to be created, even though its static initializers have not completed. Any static 
  variables in <code>A</code> that have not been initialized when the instance 
  of <code>A</code> is created will simply have their default value (0, <code>null</code>, 
  0.0, <code>false</code>, etc., depending onthe type of the variable). We can see this from the second line of output 
  produced by line 8: The variable, <code>A.b</code> is still <code>null</code>. 
  After <code>sayHi_A()</code> has completed execution we return to line 18 which 
  is responsible for the third line of output. At this point, <code>B</code>'s 
  static initializers have completed and a new instance of <code>B</code> can 
  be created created (remember, line 3 is still executing) and the <code>sayHi_B()</code> 
  method can run. Finally, the last static initializer in <code>A</code> (at line 
  4) begins executing, producing the fourth line of output.</p>
<p>To summarize, a static initializer in <code>A</code> forced <code>B</code>'s 
  static initializers to be executed. During <code>B</code>'s static initializer 
  execution, however, an instance of <code>A</code> was created. Java allowed 
  this instance of <code>A</code> to be created even though <code>A</code>'s static 
  initializer had not completed. After <code>B</code>'s static initializers completed, 
  <code>A</code>'s static initializers were resumed (even though an instance of 
  <code>A</code> had already been created by <code>B</code>'s static initializer).</p>

<p>This example illustrates  method overriding and variable shadowing. <code>static</code> methods cannot be overridden by subclasses. Even though class <code>B</code> defines a method named <code>sayHello()</code> Java does not consider this to &quot;override&quot; <code>A</code>'s method of the same name. When calling a static method, the Java compiler looks at the <i>type of the reference</i> being used to call the method&#151;it <i>does not</i> consider the type of the object <i>referred to</i> by the reference. Even though the variable <code>'a'</code> refers to an object of type <code>B</code>, because the method being called is static, Java only considers the type of the variable in deciding what method should be called. Since the variable is of type <code>A</code>, it is <code>A</code>'s static method that gets called. This accounts for the first line of output.</p>
<p>Since instance methods <i>can</i> be overridden, the Java compiler uses the object <i>referred to</i> by the variable '<code>a</code>' to determine which version of <code>sayHelloAgain()</code> will be called. To be more specific, since <code>sayHelloAgain()</code> is an instance method, the Java compiler cannot decide which version of <code>sayHelloAgain()</code> to call. Instead, method selection occurs at runtime based on the type of the object <i>referred to</i> by the '<code>a</code>' variable. This accounts for the second line of output.</p>
<p>When a subclass declares a variable of the same name as its superclass, the subclass' variable is said to <i>shadow</i> that of the superclass. Shadowed variables are different from overridden methods: When an instance method is invoked the type of the object <i>referred to</i> by the reference variable determines which method will be called, <i>not the type of the reference variable</i>. This determination occurs at runtime. When a shadowed instance variable is referred to, it the the <i>type of the reference variable</i> that determines which version of the variable is referred to, <i>not the type of object referred to by the variable</i>. This determination occurs at compile time. This accounts for the third line of output.</p>

<p>Loop #2 demonstrates the fact that a <code>finally</code> block is <em>always</em>, when control leaves a <code>try</code> block&#8212;unless the <code>try</code> block or subsequent <code>catch</code> block contains a call to <code>System.exit()</code>, which causes the VM to shutdown. The <code>break</code> statement on line 17 causes control to leave the <code>while</code> loop. In order for control to leave the <code>while</code> loop, however, it must also leave the <code>try</code> block. If a <code>finally</code> block is present, control will <em>always</em> pass to the <code>finally</code> block, regardless of what happened in the <code>try</code> block or any <code>catch</code> blocks&#8212;again, unless the <code>try</code> block or subsequent <code>catch</code> block contains a call to <code>System.exit()</code>.
