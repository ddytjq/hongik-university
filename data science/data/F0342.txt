        <p class="ArticleH2">Introduction</p>
        <blockquote>
        <p align="justify">
        <img border="0" src="../../images/A.gif" align="left">n illustrative 
        algorithm is implemented in Visual Basic, emphasizing all the four 
        fundamental aspects of distributed computing systems: resource sharing, 
        computation speedup, reliability, and communication (<a href="#1">1</a>). 
        Winsock control is used to provide effective communication and resource 
        sharing between the distributed server and its clients. Requests queuing 
        coupled to an algorithm prioritizing read over write, speed up 
        communication and preserve data integrity. System reliability is 
        enhanced by applying appropriate error handling and messaging 
        techniques. The server has direct access to a modular database. The 
        clients access data using the server's select, update and insert 
        functionality.</p>
        </blockquote>
        <p class="ArticleH2">Prioritizing Data Read over Data Write</p>
        <blockquote>
        <p align="justify">Processing simultaneous update and select requests 
        without compromising data integrity is the most crucial aspect of any 
        distributed system with multiple clients. Overcoming performance 
        bottleneck by speeding up communication is another formidable task. 
        These two challenges can be effectively addressed by implementing 
        separate queues for processing select, insert and update requests. Data 
        update is more time consuming than data select. Hence, algorithms 
        targeting to maximize performance efficiency often prioritize data 
        retrieval over table updates. This functionality segregation can be 
        easily achieved by implementing either of the algorithms outlined below 
        in pseudo-code:</p>
          <table border='0' width='80%' bgcolor='#F8F8F8' cellspacing='0' cellpadding='2' height="193">
  <tr>
    <td width='100%' height="189">
      <blockquote>
      <p class='code'><b>Algorithm 1: <br>
      </b><br>
      while (all requests are not processed) loop<br>
      <br>
      <span lang="ru">&nbsp;&nbsp; </span>if the select queue is not empty then<br>
&nbsp;&nbsp; <span lang="ru">&nbsp;&nbsp; </span>process select request<br>
      <br>
      <span lang="ru">&nbsp;&nbsp; </span>else if update queue is not empty then<br>
&nbsp;<span lang="ru">&nbsp;&nbsp;&nbsp;&nbsp; </span>process update request<br>
      <br>
      <span lang="ru">&nbsp;&nbsp; </span>else if insert queue is not empty then<br>
&nbsp;&nbsp; <span lang="ru">&nbsp;&nbsp; </span>process insert request<br>
      <br>
      end loop</p>
      </blockquote>
    </td>
  </tr>
</table>
          <table border='0' width='80%' bgcolor='#F8F8F8' cellspacing='0' cellpadding='2'>
  <tr>
    <td width='100%'>
      <blockquote>
        <pre><b>Algorithm 2:  
</b>
while (the select queue is not empty) loop 
<span lang="ru">   </span>process select request
end loop
       
while (the update queue is not empty) loop
<span lang="ru">   </span>process update request
end loop

while (the insert queue is not empty) loop
<span lang="ru">   </span>process insert request
end loop</pre>
      </blockquote>
    </td>
  </tr>
</table>
        <p align="justify">Each of the two algorithms outlined above has its 
        respective merits and drawbacks. Though only the first one has been 
        implemented, the salient features of both algorithms are discussed 
        below.</p>
        <p align="justify"><b>Algorithm 1</b> is a true read prioritized model 
        where all select queries, including those that have just arrived, get 
        processed before most pre-existing updates and inserts. In a system 
        implementing this algorithm, the lookup queue is apt to be very long, 
        especially, when read traffic is very heavy. While apparently 
        never-ending read processing continues, outstanding update and insert 
        requests are forced to wait indefinitely, and the dreaded scenario of 
        request starvation unfolds. Thus, while tremendously increasing read 
        efficiency, this algorithm severely restricts the system’s write 
        capability. If updates do not get executed in a timely fashion, lookups 
        will continue to retrieve data that is no longer valid. Thus, although 
        performance speed is maximized, this algorithm is not well suited for 
        systems where data modifications are critical.</p>
        <p align="justify"><b>Algorithm 2</b> is more amenable to real life 
        systems where equal numbers of read and write requests are expected. In 
        this model, all the three different types of requests are processed in 
        batches. After processing the queued select requests, batches of 
        pre-existing update and insert requests are processed, before honoring 
        the demands of new arrivals. Here, once control has passed over to the 
        update or insert loop, all existing write requests will get processed 
        before data lookup is resumed. Thus, data write starvation is less 
        likely to occur. As data is updated more frequently, lookups will be 
        able to provide more accurate information and the reliability of the 
        entire system will be enhanced. The main drawback is that not only 
        lookup time is increased, but the overall performance efficiency is also 
        decreased, because data update is inherently slower than data retrieval. </p>
        </blockquote>

        <p align="justify" class="ArticleH2">Error Handling for Enhancing System 
        Reliability</p>
        <blockquote>
        <p align="justify">The ability to maintain data integrity and that to 
        alert clients promptly when a connection breaks down are the two 
        fundamental requirements for ensuring the reliability of a distributed 
        computing system. We have discussed above how data validity and 
        integrity can be enhanced by using an adequate queuing algorithm. Here, 
        we will look into a few error-handling techniques that will enhance the 
        system’s reliability.</p>
        </blockquote>
        <p align="justify" class="ArticleH2">Handling Connection Errors</p>
        <blockquote>
          <p align="justify">Winsock control can be used to provide effective 
          communication between the client and the server. Two kinds of errors 
          can occur when a client attempts to connect to a server.</p>
          <ol>
            <li>
            <p align="justify">A client may try to connect to the server while 
            it is already connected. <br>
&nbsp;</li>
            <li>
            <p align="justify">The server may not be running. </li>
          </ol>
          <p align="justify">While using Winsock control, if a client tries to 
          connect when it is already connected, a connection error occurs and 
          the following warning message is displayed by the system:</p>
          <p align="center"><img border="0" src="distri1.gif"><br>
          </p>
          <p align="justify">Choosing any of the three specified options will 
          terminate the execution of the client. The client can be prevented 
          from crashing by handling this error appropriately. </p>
          <p align="justify">A connection error will also occur if a client 
          tries to connect to a server that is not running. A warning identical 
          to that specified above will be displayed by the system and the client 
          will crash if any further attempt is made to connect. If this error is 
          handled properly, the client will not crash and will be able to 
          reconnect later, when the server begins to run.</p>
          <p align="justify">An important aspect of error handling is the 
          displaying of diagnostic messages that unambiguously identify the 
          cause of the error. In several situations including those described 
          above, the system messages indicate the type of the error that has 
          occurred but do not clearly identify the underlying cause. Under such 
          conditions, the error handler should not only handle the errors 
          effectively but should also display warning messages appropriately.
          </p>
          <p align="justify">A simple method for handling the connection errors 
          described above is illustrated in the following implementation of the 
          client’s connection sub routine. In this implementation, if a client 
          tries to connect to a server while it is already connected, the 
          message, ‘Already connected’ will be displayed. When it tries to 
          connect to a server which is not running, the message, ‘Server is 
          down. Try later’ will be displayed.</p>
          <table border='0' width='80%' bgcolor='#F8F8F8' cellspacing='0' cellpadding='2'>
  <tr>
    <td width='100%'>
      <p class='code'><font color="#000080">Private&nbsp;Sub&nbsp;</font>cmdConnect_Click()&nbsp;<br>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;<br>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;<br>

&nbsp;<br>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;<br>
<font color="#000080">End&nbsp;Sub</font></p>
    </td>
  </tr>

        </blockquote>
        <p align="justify" class="ArticleH2">Handling Data Insert/Update Errors</p>
        <blockquote>
          <p align="justify">An important aspect of preserving data integrity is 
          avoiding data duplication. This is essentially done at the table level 
          in the database, by defining one or more columns to be unique 
          constraints (primary key). A runtime error will occur, if an attempt 
          is made to insert a duplicate value into a column thus constrained. 
          This error situation must be anticipated, handled and an appropriate 
          error message must be displayed. </p>
          <p align="justify">In a distributed system, in addition to inserting 
          new data, a client may also have the privilege to update existing 
          data. When a client attempts to modify data that does not exist in the 
          database, a suitable warning should be issued in order to alert the 
          user that the data must be first inserted into the database before it 
          can be updated. </p>
          <p align="justify">The following code snippets illustrate how these 
          update and insert errors can be handled in the server application. 
          They also show how appropriate warning messages can be issued to the 
          client application.</p>
          <table border='0' width='80%' bgcolor='#F8F8F8' cellspacing='0' cellpadding='2'>
  <tr>
    <td width='100%'>
      <p class='code'><font color="#000080">Private&nbsp;Function&nbsp;</font>InsertRecord(<font color="#000080">ByVal&nbsp;</font>State&nbsp;<font color="#000080">As&nbsp;String</font>,&nbsp;_&nbsp;<br>

        <p align="justify" class="ArticleH2">Conclusion</p>
        <blockquote>
          <p align="justify">An algorithm emphasizing all the four fundamental 
          aspects of distributed computing, namely resource sharing, computation 
          speedup, reliability, and communication (<a href="#1">1</a>) has been implemented in 
          Visual Basic. The entire source code for this client server 
          implementation is made available on line. Steps for obtaining the 
          source code and running the demo application are outlined below:</p>
          <ul type="square">
            <li>Name the file folders exactly as specified. <br>
&nbsp;</li>
            <li><a href="DSC.zip">Download the archive file, <b>DSC</b></a>. <br>
&nbsp;</li>
            <li>Extract its contents into a folder, C:\<b>DSC_WINSOCK</b>.&nbsp; 
            (Winzip can be instructed to automatically create this folder while 
            extracting the files from the archive, DSC.) <br>
&nbsp;</li>
            <li>Make sure that the <b>DSC_WINSOCK</b> folder has the following 
            files: <br>
            <br>
            <font face="Courier New">1. <b>State</b>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; 
            - Microsoft Access database <br>
            2. <b>Distributed Server</b> – application <br>
            3. <b>Server</b>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; 
            - Visual Basic Project <br>
            4. <b>Server</b>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; 
            - Visual Basic Project Workspace <br>
            5. <b>ServerForm</b>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; 
            - Visual Basic Form File <br>
            6. <b>Queue</b>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; 
            - class module <br>
            7. <b>Distributed Client</b> - application (recreate after modifying 
            Server IP) <br>
            8. <b>Client</b>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; 
            - Visual Basic Project file <br>
            9. <b>Client</b>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; 
            - Visual Basic Project Workspace <br>
            10.<b>ClientForm</b>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; 
            - Visual Basic Form File <br>
            11.<b>Mssccprj</b>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; 
            - Microsoft SourceSafe file <br>
&nbsp;</font></li>
            <li>
            <p align="justify">If <b>DSC_WINSOCK</b> folder is not created on the C drive, the 
            connection property of the Adodc control in the server has to be 
            modified so that the connection string will point correctly to the 
            new location of the database. If <b>DSC_WINSOCK</b> folder is 
            created on the C Drive, nothing further needs to be done to 
            establish database connection. <br>
&nbsp;</li>
            <li>Run the server by clicking open the application file, <b>
            Distributed Server</b>. <br>
&nbsp;</li>
            <li>The <b>Distributed Client</b>, application/executable file, has 
            to be recreated after changing the Server IP address to that of your 
            Host machine by following the steps outlined below: </li>
          </ul>
          <blockquote>
            <ol>
              <li>Open the Visual Basic Project File, <b>Client</b>. </li>
              <li>Modify the line, <b>sckClient.RemoteHost = &quot;127.0.0.1&quot;</b>, 
              appropriately. You can get your server’s IP address by running the 
              server application, <b>Distributed Server</b>. </li>
              <li>Create a new client executable by clicking File/Make 
              Distributed Client.exe. </li>
              <li>Name the new client application/executable, <b>Distributed 
              Client</b>. </li>
              <li>Choose yes to overwrite the old executable, when prompted.
              </li>
            </ol>
          </blockquote>
          <p></p>
          <ul type="square">
            <li>Run the client by clicking open the application file, <b>
            Distributed Client</b>. <br>
&nbsp;</li>
            <li>Try running multiple clients to see the full potential of a 
            distributed system. </li>
          </ul>
        </blockquote>
