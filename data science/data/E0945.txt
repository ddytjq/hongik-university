<h2><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">Casts<font size="3"><br>
      Difficulty: 6 / 10</font><!--mstheme--></font></h2>
<p><i>How well do you know C++'s casts? Using them well can greatly improve the
reliability of your code.</i></p>
<p align="center"><img border="0" src="../images/h-line.gif" width="248" height="2"></p>

<h3><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">Problem<!--mstheme--></font></h3>

<p>The new-style casts in (Draft) Standard C++ offer more power and safety than
the old-style C casts. How well do you know them? The rest of this problem uses
the following classes and global variables:</p>

<!--mstheme--></font><pre>    class  A             { /*...*/ };
    class  B : virtual A { /*...*/ };
    struct C : A         { /*...*/ };
    struct D : B, C      { /*...*/ };

    A a1; B b1; C c1; D d1;
    const A a2;
    const A&amp; ra1 = a1;
    const A&amp; ra2 = a2;
    char c;</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p><b><font size="4">1.</font></b>   Which of the following new-style casts are
NOT equivalent to a C cast?</p>
<blockquote>
  <p>const_cast</p>
  <p>dynamic_cast</p>
  <p>reinterpret_cast</p>
  <p>static_cast</p>
</blockquote>
<p><b><font size="4">2.</font></b>   For each of the following C casts, write
the equivalent new-style cast. Which are incorrect if not written as a new-style
cast?</p>
<!--mstheme--></font><pre>    void f() {
      A* pa; B* pb; C* pc;

      pa = (A*)&amp;ra1;
      pa = (A*)&amp;a2;
      pb = (B*)&amp;c1;
      pc = (C*)&amp;d1;
    }</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p><b><font size="4">3.</font></b>   Critique each of the following C++ casts
for style and correctness.</p>
<!--mstheme--></font><pre>    void g() {
      unsigned char* puc = static_cast&lt;unsigned char*&gt;(&amp;c);
      signed char* psc = static_cast&lt;signed char*&gt;(&amp;c);

      void* pv = static_cast&lt;void*&gt;(&amp;b1);
      B* pb1 = static_cast&lt;B*&gt;(pv);

      B* pb2 = static_cast&lt;B*&gt;(&amp;b1);

      A* pa1 = const_cast&lt;A*&gt;(&amp;ra1);
      A* pa2 = const_cast&lt;A*&gt;(&amp;ra2);

      B* pb3 = dynamic_cast&lt;B*&gt;(&amp;c1);

      A* pa3 = dynamic_cast&lt;A*&gt;(&amp;b1);

      B* pb4 = static_cast&lt;B*&gt;(&amp;d1);

      D* pd = static_cast&lt;D*&gt;(pb4);

      pa1 = dynamic_cast&lt;A*&gt;(pb2);
      pa1 = dynamic_cast&lt;A*&gt;(pb4);

      C* pc1 = dynamic_cast&lt;C*&gt;(pb4);
      C&amp; rc1 = dynamic_cast&lt;C&amp;&gt;(*pb2);
  }</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p align="center"><img border="0" src="../images/h-line.gif" width="248" height="2"></p>
<h3><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666"><a name="Solution"></a>Solution<!--mstheme--></font></h3>

<p><font color="#999933">The new-style casts in (Draft) Standard C++ offer more
power and safety than the old-style C casts. How well do you know them? The rest
of this problem uses the following classes and global variables:</font></p>

<!--mstheme--></font><pre><font color="#999933">    class  A             { /*...*/ };
    class  B : virtual A { /*...*/ };
    struct C : A         { /*...*/ };
    struct D : B, C      { /*...*/ };

    A a1; B b1; C c1; D d1;
    const A a2;
    const A&amp; ra1 = a1;
    const A&amp; ra2 = a2;
    char c;</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p><font color="#999933"><b><font size="4">1.</font></b>   Which of the
following new-style casts are NOT equivalent to a C cast?</font></p>
<p>Only dynamic_cast is not equivalent to some C cast. All other new-style casts
have old-style equivalents.</p>
<p><font color="#999933"><b><font size="4">2.</font></b>   For each of the
following C casts, write the equivalent new-style cast. Which are incorrect if
not written as a new-style cast?</font></p>
<!--mstheme--></font><pre><font color="#999933">    void f() {
      A* pa; B* pb; C* pc;

      pa = (A*)&amp;ra1;</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>Use const_cast: pa = const_cast&lt;A*&gt;(&amp;ra1);</p>
<!--mstheme--></font><pre><font color="#999933">      pa = (A*)&amp;a2;</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>This cannot be expressed as a new-style cast. The closest candidate is
const_cast, but since a2 is a const object the results are undefined.</p>
<!--mstheme--></font><pre><font color="#999933">      pb = (B*)&amp;c1;</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>Use reinterpret_cast: pb = reinterpret_cast&lt;B*&gt;(&amp;c1);</p>
<!--mstheme--></font><pre><font color="#999933">      pc = (C*)&amp;d1;
    }</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>The above cast is wrong in C. In C++, no cast is required: pc = &amp;d1;</p>
<p><font color="#999933"><b><font size="4">3.</font></b>   Critique each of the
following C++ casts for style and correctness.</font></p>
<p>First, a general note: We don't know whether any of these classes have
virtual functions, and all of the following dynamic_casts are errors if the
classes involved do not have virtual functions. For the rest of this discussion,
we will assume that the classes do all have virtual functions, making the
dynamic_casts legal.</p>
<!--mstheme--></font><pre><font color="#999933">    void g() {
      unsigned char* puc = static_cast&lt;unsigned char*&gt;(&amp;c);
      signed char* psc = static_cast&lt;signed char*&gt;(&amp;c);</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>Error: we must use reinterpret_cast for both. This might surprise you at
first, but the reason is that char, signed char, and unsigned char are three
distinct types. Even though there are implicit conversions between them, they
are unrelated, and so pointers to them are unrelated.</p>
<!--mstheme--></font><pre><font color="#999933">      void* pv = static_cast&lt;void*&gt;(&amp;b1);
      B* pb1 = static_cast&lt;B*&gt;(pv);</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>These are both fine, but the first is unnecessary since there is already an
implicit conversion from an object pointer to a void*.</p>
<!--mstheme--></font><pre><font color="#999933">      B* pb2 = static_cast&lt;B*&gt;(&amp;b1);</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>This is fine, but unnecessary since the argument is already a B*.</p>
<!--mstheme--></font><pre><font color="#999933">      A* pa1 = const_cast&lt;A*&gt;(&amp;ra1);</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>This is legal, but casting away const is usually indicative of poor style.
Most of the cases where you legitimately would want to remove the const-ness of
a pointer or reference are related to class members and covered by the 'mutable'
keyword. See <a href="006.htm"> GotW #6</a> for more discussion about const-correctness.</p>
<!--mstheme--></font><pre><font color="#999933">      A* pa2 = const_cast&lt;A*&gt;(&amp;ra2);</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>Error: this will produce undefined behaviour if the pointer is used to write
on the object, since a2 really is a const object. To see why, consider that a
compiler is allowed to see that a2 is created as a const object and use that
information to store it in read-only memory as an optimization. Casting away
const on such an object is obviously dangerous.</p>
<p>Note: I showed no examples of using const_cast to convert a non-const pointer
to a const pointer. The reason is that it's redundant; it is already legal to
assign a non-const pointer to a const pointer. We only need const_cast to do the
reverse.</p>
<!--mstheme--></font><pre><font color="#999933">      B* pb3 = dynamic_cast&lt;B*&gt;(&amp;c1);</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>Error (if you try to use pb3): since c1 IS-NOT-A B (because C is not publicly
derived from B, in fact it is not derived from B at all), this will set pb3 to
null. The only legal cast would be a reinterpret_cast, and using that is almost
always evil.</p>
<!--mstheme--></font><pre><font color="#999933">      A* pa3 = dynamic_cast&lt;A*&gt;(&amp;b1);</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>Error: since b1 IS-NOT-A A (because B is not publicly derived from A, but its
derivation is private), this is illegal.</p>
<!--mstheme--></font><pre><font color="#999933">      B* pb4 = static_cast&lt;B*&gt;(&amp;d1);</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>This is fine, but not necessary since derived-to-base pointer conversions can
be done implicitly.</p>
<!--mstheme--></font><pre><font color="#999933">      D* pd = static_cast&lt;D*&gt;(pb4);</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>This is fine, which may surprise you if you expected this to require a
dynamic_cast. The reason is that downcasts can be static when the target is
known, but beware: you are telling the compiler that you know for a fact that
what is being pointed to really is of that type. If you are wrong, then the cast
cannot inform you of the problem (as could dynamic_cast, which would return a
null pointer if the cast failed) and at best you will get spurious runtime
errors and/or program crashes.</p>
<!--mstheme--></font><pre><font color="#999933">      pa1 = dynamic_cast&lt;A*&gt;(pb2);
      pa1 = dynamic_cast&lt;A*&gt;(pb4);</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>These two look very similar. Both attempt to use dynamic_cast to convert a B*
into an A*. However, the first is an error while the second is not.</p>
<p>Here's the reason: as noted above, you cannot use dynamic_cast to cast a
pointer to what really is a B object (and here pb2 points to the object b1) into
an A object, since B inherits privately, not publicly, from A. However, the
second cast succeeds because pb4 points to the object d1, and D does have A as
an indirect public base class (through C), and dynamic_cast is able to cast
across the inheritance hierarchy using the path B* -&gt; D* -&gt; C* -&gt; A*.</p>
<!--mstheme--></font><pre><font color="#999933">      C* pc1 = dynamic_cast&lt;C*&gt;(pb4);</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>This too is fine, for the same reason as the last: dynamic_cast can navigate
the inheritance hierarchy and perform cross-casts, and so this is legal and will
succeed.</p>
<!--mstheme--></font><pre><font color="#999933">      C&amp; rc1 = dynamic_cast&lt;C&amp;&gt;(*pb2);
  }</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">

<p>Finally, this isn't fine... since *pb2 isn't really a C, dynamic_cast will
throw a bad_cast exception to signal failure. Why? Well, dynamic_cast can and
does return null if a pointer cast fails, but since there's no such thing as a
null reference it can't return a null reference if a reference cast fails.
There's no way to signal such a failure to the client code besides throwing an
exception, so that's what the standard bad_cast exception class is for.</p>
