NEW_SUBJECT:  How-To Java: "Zip your data and improve the performance of your network-based applications" by Todd Sundsted <P>
READ_ME:  /jw-11-1998/jw-11-howto.html <P>
<STRONG> WinZip doesn't deliver </STRONG><P><P>

Todd,
<P>
I was just working with Java's zip classes and  hit a snag.  I have created an archive which, when unzipped with an old command-line version of  PKUNZIP (version 1.1), works as expected.  However, when I try the same with WinZip 6.2 (the version Sun recommends), I get a collection of empty files. Examining the archive with WinZip 6.2 shows that the files are there and that WinZip knows their respective sizes. What could be going wrong? <P>
Steven Primrose-Smith <P>
<FONT COLOR="#006600">  Steven, <P>
I'm assuming you've created the zip file from Java. <P>
Here are some thoughts: <P>
First, information about a zip file entry is stored in (potentially) two places.  It may or may not be stored with each entry, and it is stored at the end of the zip file.  WinZip's explorer-like screen listing gets its information (I suspect) from the directory at the end of the zip file.  That's why the zip file looks okay. <P>
What's interesting is that both WinZip and Java use zlib (I believe) whereas PKUNZIP uses its own library.  The InfoZIP tools also use zlib -- you might want to give them a try to see if they do any better.  Also, determine whether or not WinZip can extract entries from jar files created with Sun's jar tool -- jar files are also zip files. <P>
Todd Sundsted :</FONT> <P>
LETTER_DIV:  <STRONG> Weak zip implementation? </STRONG><P><P>
Todd, <P>
I'm new to compression field and would like to ask if there is a way to avoid reading the <code>FileInputStream</code> twice?  (as in your example). <P>
Damir Kirasic <P>
<FONT COLOR="#006600">
<P>
Damir, <P>
You've found an area where the zip implementation is, in my opinion, weak.  Here's the deal: <P>
The CRC value is only <em>necessary</em> when writing a STORED entry. The <code>ZipInputStream</code> class needs the value so that it can determine where the data ends.  The <code>ZipInputStream</code> class can determine where DEFLATED data ends without the CRC value. <P>
You might ask why it uses this value and not the uncompressed size of the file (which is stored in the zip header) to determine when to stop reading.  Unfortunately, the <code>ZipOutputStream</code> class doesn't reliably record this value.  Apparently, it assumes that all zip data is coming from someplace like <code>System.in</code> (and is, therefore, of unknown length).  Consequently, it doesn't record the value for size you set with <code>ZipEntry.setSize()</code>.  So even if you <em>know</em> the size of the file you want to add to the zip file, you can't count on it using that value. <P>
Anyway, I read the <code>FileInputStream</code> the first time to calculate the CRC value.  I read it the second time to write the data. If you always write DEFLATED entries you don't need calculate the CRC value, but it doesn't hurt either.  I wanted to provide code that worked in either case. <P>
Todd Sundsted :</FONT> <P>
NEW_SUBJECT:  How-To Java: The Mobile Agents series <P>
READ_ME:  /jw-06-1998/jw-06-howto.html READ_ME:  /jw-07-1998/jw-07-howto.html READ_ME:  /jw-09-1998/jw-09-howto.html READ_ME:  /jw-10-1998/jw-10-howto.html <P>
<STRONG> Prolog: museum piece or cutting edge?  </STRONG><P><P>
Todd, <P>
I read your article ("Agents can think, too!" <em>JavaWorld</em>, October 1998) and found it very thought-provoking. I'm considering doing some research in speech recognition and it would appear that the AI language of choice for this area is Prolog. How good is Prolog in the overall scheme of things? Is it doomed to be a museum piece in the same fashion that you suggest Lisp will be? In short, does it make sense to invest time and money in learning it? <P>
And lastly, would you know offhand which are the best universities in the US that offer undergraduate and postgraduate programs in speech-recognition technologies? <P>
James N. Carmichael <P>
<FONT COLOR="#006600">  James, <P>
A professor of mine once declared, "Prolog is a great one-idea language." He was, I think, referring to the fact that Prolog is great a what it does, but it only does one thing really well.  I once tried to write a stack based (RPN) calculator in Prolog (don't ask why).  I managed to finish it, but it took far longer and was far more confusing than it would have been had I written it in C (or Lisp). <P>
You need to pick the best language for the problems you're trying to solve.  C, C++, and Java are popular because they make it easy to write the types of applications a lot of people want to write.  Lisp and Prolog make good AI languages, and I suspect they'll both (especially Lisp) be around until someone creates something better.  It would have been a mistake to write the broker/buyer/seller code in Java. <P>
With that said, I think you'll find the area of speech recognition to be something of a hybrid.  Depending on the level at which you work, you may use Lisp and you may use C.  It will depend on whether you're studying the higher level cognitive processing, or the low level auditory processing.  You might even wind up building silicon (Richard Lyon and Carver Mead built a model of the cochlea in silicon). <P>
As to recommendations: unfortunately, they wouldn't be much more than guess work.  I'd suggest reading the publications and contacting the people doing the work you think is interesting. <P>
Todd Sundsted <P>
:</FONT> <P>
LETTER_DIV:  <STRONG> And another thing... </STRONG><P><P>

Todd, <P>
I enjoyed your piece on Jess ("Agents can think, too!" <em>JavaWorld</em>, October 1998); it was well written and easy to follow. <P>
However, I do a have a few suggestions.  First, you should have included the URL for Jess. <P>

John, <P>

<BR>

&nbsp;&nbsp; }<BR>

As you can see the bitmap header and the information header are the same. The difference is in the color table that follows the information header. The color table contains 256 entries of RGBU (U for unused). In this table you set your gray scale colors. Next, you will have to compute the start of the bitmap data using the following formula: <code>startOfBitmap = sizeOfBitmapHeader + sizeOfInfoHeader + (256 * 4)</code>.  And remember, each entry in the bitmap data is an index into the color table.  Because there are only 256 colors, each bitmap data is 1 byte, so you will reduce your file size considerably from a 24-bit format. <P>
To conclude, you will have to scan in your memory image and create a color table, store that color table after the bitmap info header, and store the image itself, as each byte is an index in the color table. <P>

NEW_SUBJECT:  "Merging Java and Win32: A new way to develop Windows applications" by Geoff Friesen <P>
READ_ME:  /jw-07-1998/jw-07-java-win32.html <P>
<STRONG> Getting a C++ header file with Java </STRONG><P><P>
Geoff, <P>
I'm quite interested in combining Java with C++, and your article was a big help. <P>
Everything seems fine until I execute <code>java useSysInfo</code>. Once I do this, a <code>UnsatisfiedLinkError</code> is thrown in the line where the call to the native method is done (<code>System.out.println ("PATH = " + SysInfo.getenv ("PATH") + "\n");</code>). <P>
I think I have done everything right, but I am not sure how you get the sysInfo.h file.
<P>
I'm hoping you can shed some light on this question. <P>
Javier Finez <P>
<FONT COLOR="#006600">
<P>
Javier, <P>
You use the javah tool to generate a C++ header file that contains the C++ JNI-style function prototypes for those Java methods defined with the native attribute.  Don't forget to include  the <code>-jni</code> option with javah; otherwise, the header file will not be generated.  Here is an example: <P>
<CODE><P>
<BR>

public class SysInfo<BR>

{<BR>

&nbsp;&nbsp; static<BR>

&nbsp;&nbsp; {<BR>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;try<BR>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;{<BR>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; System.loadLibrary ("SysInfo");<BR>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;}<BR>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;catch (java.lang.UnsatisfiedLinkError e)<BR>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;{<BR>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;}<BR>

&nbsp;&nbsp; }<BR>

<BR>

&nbsp;&nbsp; public static native String getenv (String evName);<BR>

}<BR>

<BR>

javah -jni SysInfo<BR>
</CODE><P>
This example will create a file called SysInfo.h.  The C++ JNI-style function prototype is: <P>
<code>JNIEXPORT jstring JNICALL Java_SysInfo_getenv (JNIEnv *, jclass, jstring);</code> <P>
As to the <code>UnsatisfiedLinkError</code>, this will occur under the following circumstances. <P>
<OL> <LI>The DLL (Windows) or shared library (Unix) cannot be found.  This DLL or<P>
library must be located in the current path. <P>
<LI>The C++ function stored in the DLL or shared library has a name that does<P>
not match the name found in the header file. 
<P>
</OL> <P>
Geoff Friesen <P>
