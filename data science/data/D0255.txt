Another problem is the lack of some important functionality in the JVM needed for high performance computing and integration with legacy code.<br/><br/>My assessment is that most of the server-side APIs people really use are available in open implementations. The main headache in producing a Java2 implementation is Sun's massive (and IMO not so well thought out) client, gaming, and media libraries.<br/><br/>So, maybe the thing to do is first to identify what freely available libraries constitute core server functionality and define an open platform in terms of those. Sun's client side libraries can then be replaced with native bindings to open, cross-platform toolkits (which would also address Swing's performance problems).<br/><br/>Furthermore, a Java-to-CLR compiler is pretty simple and would allow people to freely recompile their code for either platform. If there is a well-defined set of open Java libraries, the same libraries could be available both on the JVM and on the CLR. And if you run on the CLR, you additionally get access to value classes and simpler binding to native code if you like.<br/><br/>Of course, this would never appeal to the crowds who like to seek salvation in pledging undying loyalty to a single vendor, be it Sun or Microsoft (the mentality is the same). Sun would doubtlessly also not like it one bit, but as long as you don't call the result &quot;Java&quot;, there isn't much they could do about it.<br/><br/>I think this could be a viable path for the open source community, as well as some of the smarter commercial software developers, to achieve independence from Sun corporate politics and to make sure that the efforts of both Java and C# developers benefit each other.<BR>
