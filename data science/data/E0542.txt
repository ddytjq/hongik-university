<CENTER><H3><FONT COLOR="#AOAO99">
Connection Pipe Classes
</FONT></H3></CENTER><HR>

<!-- Author and contact details -->

This article was contributed by 
<A HREF="mailto:iont@hotmail.com">Ion Tichy</A>.<BR>
<A HREF="http://www.iont.20m.com" target="new">

<!-- Sample image - gif or jpg -->
<!-- For which environment is this H2 specific??? -->
<p><u>Environment:</u> VC6, WindowsNT4, Windows2000

<P><IMG SRC="ConnPipeClient.gif" alt="Client">
<P><IMG SRC="ConnPipeServer.gif" alt="Server">

<p><H2>1.  Purpose</H2>
<p>
<br>Simple and fast communication between all types of MFC Applications on a single or multiple computers in Windows NT. 
<p><H2>2.  Main features</H2></p>
<p>
<br><LI>Data transmission from Clients to Server, their processing and results return, if needed.
<br><LI>Data transmission from Server to Clients informing them about any event.
<br><LI>Different schemes of buffer allocations.
<br><LI>Automatic Server loading.
<br><LI>Automatic Server termination without Clients, if needed.
<br><LI>Utilities for registration, automatic loading, tracing and Wizard. 
</p>

<p><H2>3.  Introduction</H2></p>

<br>Several years ago I wrote classes by using Mailslots for communication between Clients and Server. According to the project requirements these classes had to worked in Windows 95. In spite of numerous problems and limitations of Mailslots these classes work properly in Windows 95 and Windows NT. 

<p>The above-mentioned problems are the following: </p>

<ul>
<li>8-bytes limitation for Mailslots names in Windows 95 and </li>
<li>numerous data transmission across Mailslots between Computers, if they have several identical installed communiciation protocols. </li>
</ul>

<p>Microsoft considers this as "by design". In other words, it is not an error, but I don't agree.</p>

<p>In these classes Server has the receiving thread and Clients send data with identificator of Client (Computer name and Client name) to this thread. It increases summary data that Clients transmit across Mailslots and it is the disadvantage of this method.</p>

<p>I took a course of COMs and ATL. After that it was evident that there were very many undocumented things in that technology. ATL is a convenient tool, but I have discovered several limitations, e.g. it is impossible to make ATL-Server from the existing MFC application.  </p>

<p>DCOM implementation needs great efforts too. For example, I wrote my first program with DCOM in  almost 2 weeks and had to ask Microsoft for numerous consultations. That's why in spite of my wish to use COM/DCOM in my new project I had to give up this idea.</p>

<p>The new project that I have begun to work at had high requirements for the speed of data processing. This factor had to be taken into account in the design of the new communication tool. Fortunately, unlike COM, there were no requirements to the design of the universal tool and this made the development easier. </p>

<p>For the new project I have chosen the protocol Named Pipes working only in Windows NT. In literature this protocol is defined as following:</p>
<blockquote>
An inter-process communication mechanism that allows one process to communicate with another local or remote process. </blockquote>

<p>I have tried to implement the ideas that I used in my first project with Mailslots as well as the ideas of COM. In the next chapter you'll learn the Client-Server architecture, based on the Named Pipes protocol.</p>

<p><H2>4.  Client-Server Architecture</H2></p>

<p>There are following variants of connections Client and Server. </p>
<ul>
<li>Client and Server are on a single computer. 
<li>Client and Server are on  multiple computers.
</ul>


<p><a href="ConnPipeSchemes.gif" target=newFrame>
<br><img src="ConnPipeSchemes.gif" width=450>
<br><font size=3><em>Click here for larger image</em></font>
</a> </p>

<p>When Server is loaded, Client tries to get into connection with Server.</p>

<p>If Server is not loaded and Client and Server are on a single computer, Client gets information about Server from Registry of this computer(see Computer 4). </p>

<p>If Server is not loaded and Client and Server are on multiple computers, Client gets information about Server from Registry of the remote Server computer across utility AgentCP (see Computers 1 and 2). This program searches Server information in Registry of Server Computer and loads it.</p>

<p>As seen from Figure 1, for successful functioning  of  the above-mentioned process it is necessary:</p>
<ul>
<li>To write information about Server in Registry of Server Computer. </li>
<li>To run utility AgentCP on the remote computer previously.</li>
</ul>

<p>Thus the Server automatic loading is carried out. Process-Server may be terminated automatically, if Clients are not connected.</p>
