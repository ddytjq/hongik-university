<CENTER><H3><FONT COLOR="#AOAO99">
Template Mania
</FONT></H3></CENTER><HR>

<!-- Author and contact details -->

This article was contributed by 
<A HREF="mailto:aliakseis@yahoo.com">Aliaksei Sanko</A>.

<!-- For which environment is this code specific??? -->
<p><u>Environment:</u> VC6 SP5, NT4 SP3, Windows 98, Windows 2000

<p>
There is an interesting article concerning template-based technique on www.cuj.com <a href="#1">[1]</a>. A simple struct is introduced there that helps to implement dummy arguments converting values to distinquishable types:
<PRE>
template&lt;int v&gt;
struct Int2Type
{
   enum { value = v };
};
</PRE>

The main article idea is generally described in terms of classes, but there is a reason why it can be especially useful for template functions with VC. There is a bug Q240871 in the Microsoft Visual C++ 6.0 compiler, its description from the MSDN is given below:

<p>
<i>
SYMPTOMS<P>
If all the template parameters are not used in function arguments or return type of a template function, the template functions are not overloaded correctly.<p>
CAUSE<P>
The bug is the result of the way the compiler decorates template function names. Name Decoration uses the arguments and return type and doesn't use the explicitly specified template argument type. Therefore, all three template function instantiations receive the same decorated name&lt;...&gt;
<P>
RESOLUTION<P>
Use dummy arguments to the function.
</i>
<P>

There is another interesting technique outlined in <a href="#2">[2]</a>. Templates do not deal with objects hierarchy, but dummy arguments combined with ellipses make it possible to specialize templates for classes and their subclasses. Let us implement a pointer validity checking utility based on these principles as an example.
<P>
MFC has two validating macros: ASSERT_VALID for CObject-based classes and ASSERT_POINTER for others. We introduce a single template function that supersedes both of them. The whole stuff being workable, some details will be omitted for simplicity. ASSERT_VALID macro invokes AfxAssertValidObject(). Among other things, it checks the object virtual table pointer. To do so, we need to find out if the class has one first. The next accessory struct can help us:
<PRE>
template&lt;class T&gt; struct HasVirtualTable
{
   class X : public T
   {
      X();
      virtual void dummy();
   };
   enum { has_table = sizeof(X) == sizeof(T) };
};
</PRE>
Then we introduce an AssertVTable template function for the "TRUE" and its void specialization for the "FALSE" (0) Int2Type template parameter:
<PRE>
template&lt;typename T&gt; inline void AssertVTable(const void* pData, T)
{
  ASSERT(AfxIsValidAddress(*(void**)pData, sizeof(void*), FALSE));
}

inline void AssertVTable(const void*, Int2Type&lt;0&gt;) {}
</PRE>
Here goes the common validating routine with an ellipse at the end of the arguments list:
<PRE>
template&lt;class T&gt; inline void AssertValidPointer(const T* pData, ...)
{
  ASSERT(AfxIsValidAddress(pData, sizeof(T)));
  AssertVTable(pData, 
               Int2Type&lt;HasVirtualTable&lt;T&gt;::has_table&gt;());
}
</PRE>
The next validating routine substitutes the previous one for CObject subclasses:
<PRE>
template&lt;class T&gt; inline void AssertValidPointer(const T* pOb,
                                                      const CObject*)
{
  ASSERT(AfxIsValidAddress(pOb, sizeof(T)));
  ASSERT(AfxIsValidAddress(*(void**)pOb, sizeof(void*), FALSE));
  pOb-&gt;AssertValid();
}
</PRE>
We can add customized validating routine for any class hierarchy branch in the same way.
<p>
Here are the top level functions to be used in the application:
<PRE>
template&lt;class T&gt; inline void CHECK_ADDRESS(const T* pData)
{
  AssertValidPointer(pData, pData);
}
template&lt;class T&gt; inline void CHECK_NULL_OR_ADDRESS(const T* pData)
{
  if(NULL != pData)
     AssertValidPointer(pData, pData);
}
</PRE>
We put stuff listed above into the #if/#else/#endif brackets to disable the functionality in the release build:
<PRE>
#ifdef _DEBUG
<span class="codeComment">   ...</span>
#else <span class="codeComment">//_DEBUG</span>
#define CHECK_ADDRESS(pData)          ((void)0)
#define CHECK_NULL_OR_ADDRESS(pData)  ((void)0)
#endif <span class="codeComment">//!_DEBUG</span>
</PRE>
Now we can use the CHECK_ADDRESS() and CHECK_NULL_OR_ADDRESS() functions.
<p>
At the end of the article I would dare to propose an exotic way to simulate enums using templates. It was written for fun only, but seems to be interesting and can actually help when there is no other way to keep enum items together:
<PRE>
#include &lt;iostream&gt;

#define ENUM(val) \
template&lt;&gt; struct cnt&lt;__LINE__&gt; { enum {v = cnt&lt;__LINE__-1&gt;::v+1}; }; \
enum { val = cnt&lt;__LINE__&gt;::v };	\
template&lt;&gt; struct flg&lt;val&gt; { enum {v = 0}; };

namespace test
{
  template&lt;int i&gt; struct flg { enum {v = -1}; };
  template&lt;int i&gt; struct cnt { enum {v = cnt&lt;i-1&gt;::v }; };
  template&lt;&gt; struct cnt&lt;__LINE__&gt; { enum {v = -1 }; };

  ENUM(a)
<span class="codeComment">
// Some stuff here
</span>
  ENUM(b)
<span class="codeComment">
// Some stuff here
</span>
  ENUM(c)

  template&lt;int i&gt; struct bnd 
  {
     enum 
     {
        u = bnd&lt;(i+1) | flg&lt;i+1&gt;::v&gt;::v,
        v = (-1 == u)? i : u
     };
  };
  template&lt;&gt; struct bnd&lt;-1&gt; { enum { v = -1 }; };
  enum { upper_bound = (bnd&lt;0&gt;::v | flg&lt;0&gt;::v)+1 };
};

int main()
{
  cout &lt;&lt; test::a &lt;&lt; ',' 
       &lt;&lt; test::b &lt;&lt; ',' 
       &lt;&lt; test::c &lt;&lt; ',' 
       &lt;&lt; test::upper_bound &lt;&lt; "\n";
  return 0;
}
</PRE>
