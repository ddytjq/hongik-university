  <H3><FONT COLOR="#AOAO99"> MemCheck Driver Memory Tool</FONT></H3>
</CENTER><HR>

<!-- Author and contact details -->
This article was contributed by <A HREF="mailto:feedback@L5sg.com">L5 Software 
Group, LLC</A>. 
<!-- Sample image - gif or jpg -->

<P><u>Environment:</u> Windows NT/2K/XP driver development 
<p> 
<p><font size="4">Purpose</font>
<hr>
<p>The MemCheck code is designed to provide Windows NT/2K/XP driver developers 
  with a tool to help in the detection of the following memory handling issues:</p>
<p> </p>
<p>Buffer overrun <br>
  Buffer corruption <br>
  Buffer use after buffer release <br>
  Double buffer releases <br>
</p>
<p> <font size="4"><br>
  MemCheck&#146;s Routines<br>
  </font></p>
<hr>
<font size="4"> </font>
<p>MemCheck has a specific API (set of functions) that should be used in place 
  of existing routines. </p>
<table width="75%" border="1">
  <tr> 
    <td><b>Original routine<br>
      </b></td>
    <td><b>MemCheck routine</b></td>
  </tr>
  <tr> 
    <td>ExAllocatePool(&#133;)<br>
    </td>
    <td>MEM_ALLOCATE(&#133;)</td>
  </tr>
  <tr> 
    <td>ExAllocatePoolWithTag(&#133;)<br>
    </td>
    <td>MEM_ALLOCATE_WITH_TAG(&#133;)</td>
  </tr>
  <tr> 
    <td>ExFreePool(&#133;)<br>
    </td>
    <td>MEMCHECK_FREE(&#133;)</td>
  </tr>
</table>
<p>The parameters for each of the MemCheck routines are exactly the same as those 
  in the original (DDK) functions. By replacing the original calls with the memCheck 
  versions, validation can be done during the allocation and release routines. 
</p>
<p> </p>
<p>In addition to the functions listed above, MemCheck also has the following 
  routines:</p>
<p> </p>
<table width="75%" border="1">
  <tr> 
    <td><b>MemCheck Routine</b></td>
    <td><b>Description</b></td>
  </tr>
  <tr> 
    <td>MEM_CHECK_INIT()</td>
    <td>Initialize memCheck's resources. This routine should be called before 
      any other memCheck routines are used. </td>
  </tr>
  <tr> 
    <td>MEM_CHECK()</td>
    <td>Forces memCheck to validate all buffers being monitored.</td>
  </tr>
  <tr> 
    <td>MEM_LIST_DISPLAY()</td>
    <td><br>
      Displays a list of all buffers being monitored by memCheck. In addition, 
      each buffer being monitored is validated for correctness (the header and 
      footer tags are checked). </td>
  </tr>
  <tr> 
    <td>MEM_VALID_ACCESS()</td>
    <td>Tests ptr to see if it points to a location within a valid buffer being 
      monitored by memCheck. </td>
  </tr>
  <tr> 
    <td>MEM_CHECK_EXIT()</td>
    <td>Frees resources used by memCheck. This routine should be called when unloading. 
      Make sure not to make further calls to memCheck routines after calling MEM_CHECK_EXIT() 
    </td>
  </tr>
</table>
<p><br>
</p>
<p> </p>
<p> </p>
<p><font size="4">MemCheck Settings</font></p>
<p> </p>
<hr>
<p>The settings for MemCheck are contained in the memcfg.h file. Each of these 
  settings is a simple macro. By defining and undefining a macro, the setting 
  can be turned on or off.</p>
<table width="75%" border="1">
  <tr> 
    <td> 
      <p> </p>
      <p><b>MemCheck Setting</b></p>
    </td>
    <td><b>Description</b></td>
  </tr>
  <tr> 
    <td>MEMCHECK_ON</td>
    <td>Enable/disable the entire memcheck package.</td>
  </tr>
  <tr> 
    <td>MEMCHECK_DEBUGPRINT</td>
    <td><br>
      Enables displaying of MEMCHECK_DEBUG() macro information this is mostly 
      debug information for memCheck itself. </td>
  </tr>
  <tr> 
    <td>MEMCHECK_DISPLAYPRINT</td>
    <td><br>
      Enables displaying of MEMCHECK_PRINT() macro information</td>
  </tr>
  <tr> 
    <td>MEMCHECK_DISPLAY_FREES</td>
    <td><br>
      Display information when buffer is freed</td>
  </tr>
  <tr> 
    <td>MEMCHECK_DOUBLE_FREES</td>
    <td>Enable checking for double frees...this causes extra memory to be used 
      and should NOT be left enabled for a shipping product. </td>
  </tr>
  <tr> 
    <td>MEMCHECK_HALT_ON_BAD</td>
    <td>Causes a debug breakpoint to occur when a bad buffer tag is encounted</td>
  </tr>
  <tr> 
    <td>MEMCHECK_FREE_ON_EXIT</td>
    <td>Allows memcheck to free its resources upon exit. This is normally left 
      on</td>
  </tr>
</table>
<p><font size="4"><br>
  <br>
  How MemCheck works</font></p>
<p> </p>
<hr>
<p>MemCheck&#146;s main functionality is exercised when all allocation and release 
  routines (ExAllocatePool(&#133;), ExAllocatePoolWithTag(&#133;) and ExFreePool(&#133;)) 
  are replaced with the MemCheck analog (see MemCheck&#146;s Routines above for 
  more information). With these routines in place, MemCheck maintains a list of 
  each buffer allocated.  When the client requests a 1K buffer be allocated, MemCheck actually allocates 
  a larger buffer. This extra space is used to store tags at the start (header) 
  and end (footer) of the buffer.  The client is returned a pointer to a buffer of the requested size (assuming 
  the call was successful). MemCheck then writes a specific value in the header 
  and footer tags. If at any point MemCheck realizes that the tags are invalid, 
  it can tell that the user has perturbed their memory. </p>
<p> </p>
<p><b>NOTE: MemCheck stores more information within the buffer than just the tags 
  and client data. The above description is a simplification of the actual method 
  MemCheck uses. Please refer to memcheck.c code to see how the real buffer list 
  is maintained.</b></p>
<p> </p>
<p>Another useful feature of MemCheck is MEM_VALID_ACCESS(). By calling MEM_VALID_ACCESS(ptr), 
  MemCheck can compare the ptr address to the addresses of each of the monitored 
  buffers. If the address is not within a valid buffer range, MemCheck can display 
  a warning. </p>
<p> </p>
<p>When the MEMCHECK_DOUBLE_FREES macro is defined (in memcfg.h) and the client 
  frees a buffer, the buffer is not actually released. Instead, it&#146;s information 
  is maintained. In the event that an attempt is made to free the same ptr, MemCheck 
  will be able to notify the client. </p>
<p></p>
<p></p>
<p><br>
</p>
<p> </p>
<p><font size="4">Using the MemCheck Code</font></p>
<p> </p>
<hr>
<p>The first step in using the MemCheck code is to call MEM_CHECK_INIT() when 
  your driver initializes and MEM_CHECK_EXIT() when your driver unloads. The next 
  step is to replace all the original DDK memory allocation and release calls 
  with the MemCheck counterparts <i>(see MemCheck&#146;s Routines above for more 
  information)</i>. After completing these steps, make sure MemCheck is enabled 
  (via the MEMCHECK_ON macro in memcfg.h). Then build the driver and run it. MemCheck 
  will be able to scour the memory usage right away and try to find problems.</p>
<p> </p>
<p>In addition to the previous steps, users may find it useful to use the other 
  &#145;helper&#146; functions to track down buffer problems. These functions 
  are</p>
<p> </p>
<ul>
  <li> MEM_CHECK()</li>
  <li>MEM_LIST_DISPLAY()</li>
  <li>MEM_VALID_ACCESS()</li>
</ul>
<p> </p>
<p><i>(see MemCheck Routines above for a description of these routines).</i></p>
<p> </p>
<p>When you have finally tracked down all memory issues, you can disable the MemCheck 
  package by undefining the MEMCHECK_ON macro in memcfg.h. This has the effect 
  of making your code build with no code from MemCheck being included. This means 
  that the size and performance of the driver will be the same as one using only 
  the DDK calls.</p>
