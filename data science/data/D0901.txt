<font color="#336699" face="Arial,Helvetica,Sans-serif"><P>
<strong> C# vs. Java </strong><br></font><P>
<strong><a href="/javaworld/jw-11-2000/jw-1122-csharp1.html"><font size="+1">'C#: A language alternative or just J--?, Part 1'</font></a></strong><br><P>
<em>Mark Johnson </em><br><P>

<STRONG> Prescription for the C# headache</STRONG><P><P>
Mark,<P>

Regarding your question and answer:<P>

"Q. What's more of a headache than a bug in a compiler?<P>

A. Bugs in six compilers."<P>

I don't think that anyone working on C# will compile code in six compilers. As for the Microsoft specification, developers will compile code in one compiler, but six other compilers will use that code. Consider <code>msxml.dll</code> "XML DOM interface"; it is created in C++. If C++ can use that, developers will have to create one component for VB, one for VJ, one for Delphi, and so on. And that will be an additional headache.<P>

Change is a sign of development.<P>

Urvesh Patel<P>

<FONT COLOR="#006600"><P>

Mr. Patel,<P>

Will you agree that programmers might use four languages instead of six, since you name four yourself: C++, VB, Delphi, VJ?<P>

Maybe I need to clarify the point I was trying to make. Your point (if I understand it correctly) is that code written in one language can be packaged into a <code>dll</code> and used in other languages. I agree that's a good thing, and that's not the concern I have about .Net and the "any-language-except-Java" approach Microsoft is taking. See "<a href="http://www.computerweekly.co.uk/cwarchive/daily/20001123/cwcontainer.asp?name=C3.HTML&SubSection=6&ct=daily">Microsoft Tells Java Converts -- Don't Bother</a>" (<em>ComputerWeekly.com,</em> November 2000) if you still think Microsoft will try to support Java in any way.<P>

Anders Hejlsberg's view (see O'Reilly's "<a href="http://windows.oreilly.com/news/hejlsberg_0800.html">Deep Inside C#:<P>
An Interview with Microsoft Chief Architect Anders Hejlsberg</a>," by John Osborn) is that programmers should be free to work in whatever language they wish, and the platform should allow languages to interoperate. Classes defined in one language can be subclassed in another language, and further subclassed in a third language. So programmers won't just be writing C# programs and using other languages' modules. They could potentially write modules in all of those languages (each with its own compiler), and cross-link them (via MSIL) at runtime.<P>

That may sound cool to programmers, but programmer preferences are only part of the big picture of the software lifecycle. I think the value that .Net's multiple-language capability provides hardly makes up for the complexity it introduces. I'm not convinced that it's good for code stability, clean design, and especially maintainability. Here are some reasons why:<P>

<ul><P>
<li><strong>Maintainability:</strong> Code testers and maintainers will have to be fluent in every language programmers choose.</li><P>

<li><strong>Teamwork:</strong> Programmers will have to spend extra time training each other in different languages as they collaborate on design and implementation.</li><P>

<li><strong>Hiring and training:</strong> "Sorry, boss, our only Haskell programmer quit today, and a quarter of our app is written in Haskell. But don't worry, I bought a book. We'll put out another release in 6 months or so."</li><P>

<li><strong>Lock-in:</strong> .Net is a single-vendor platform. While you're deciding where you want to go today, think about who decides where you'll go tomorrow.</li><P>

<li><strong>Compiler version compatibility:</strong> Will there be compatibility problems between versions of the compilers you're using? If so, just work it out with the compiler vendors: all "n" of them.</li><P>

<li><strong>Debuggability:</strong> Do you think popping back and forth between languages as methods call inherited methods will make debugging easier?</li><P>

</ul><P>

I believe that the multiple language platform that .Net touts sounds good -- as long as you don't give it too much thought. I think organizations that follow that route will eventually find themselves in an unmanageable morass of code chunks written in different languages; essentially, a "<a href="http://www.laputan.org/mud/mud.html">Big Ball of Mud</a>," full of chunky little code pieces that bear no comprehensible relation to one another.<P>

Having said this, I could certainly be wrong. I've been a big fan of Tcl (a lightweight, embeddable scripting language) for years, and have always liked embedded interpreters, command patterns, and the like. And .Net will probably help to create systems that cleanly partition language responsibilities. But I wouldn't bet any folding money that most designs will be that clean. For the most part, I believe .Net will give designers just enough rope to hang themselves.<P>

Not all change is progress.<P>

Mark Johnson<P>
:</FONT><P>

<STRONG> Why use Vector and not the Collection API?</STRONG><P><P>
Mark,<P>

First, I must say that your article is excellent and gives good insight into the differences and similarities in C# and Java. I'm looking forward to future articles.<P>

However, I was disappointed with your comparison of iterating in the two languages. I like Java a lot, but have never been satisfied with the use of <code>Vector</code>. Therefore, I was very happy with the Collection API -- I'm sure you were too. So why didn't you chose to use <code>ArrayList</code> and <code>Iterator</code>? This makes the power of the Collection API clear, instead of using a special language construct as in C#.<P>

To conclude, I would replace:<P>

<CODE><P>
<BR>
import java.util.Vector;<BR>
public static int sum(Vector v) {<BR>
&nbsp;&nbsp;&nbsp;&nbsp;int sum = 0;<BR>
&nbsp;&nbsp;&nbsp;&nbsp;for (int j = 0; j < v.size(); j++) {<BR>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; int i = (Integer)v.elementAt(j);<BR>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;sum = sum + i.intValue();<BR>
&nbsp;&nbsp;&nbsp;&nbsp;}<BR>
&nbsp;&nbsp;&nbsp;&nbsp;return sum;<BR>
}<BR>

</CODE><P>
with:<P>

<CODE><P>
<BR>
import java.util.ArrayList;<BR>
public static int sum(ArrayList list) {<BR>
&nbsp;&nbsp;&nbsp;&nbsp; int sum = 0;<BR>
&nbsp;&nbsp;&nbsp;&nbsp; Iterator iterator = list.iterator();<BR>
&nbsp;&nbsp;&nbsp;&nbsp; while(iterator.hasNext()) {<BR>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; Integer value = (Integer) iterator.next();<BR>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; sum = sum + value.intValue();<BR>
&nbsp;&nbsp;&nbsp;&nbsp; }<BR>
&nbsp;&nbsp;&nbsp;&nbsp; return sum;<BR>
}<BR>

</CODE><P>
This certainly looks cleaner and gives Java a fair chance. The <code>for</code> construct should be compared with the C# variant as well.<P>

Martin Bravenboer<P>

<FONT COLOR="#006600"><P>

Martin,<P>

I wish all questions were so easy. The reason I used <code>Vector</code> instead of the Collection was simply force of habit. I use the Collection API all the time, but for some reason (maybe the late hour), when I wrote the comparison, I reverted to using <code>Vector</code>. <P>

Your solution is definitely superior; it demonstrates that anything done in a language can often be done (almost) as easily from outside the language, and provides one more piece of evidence that I need more sleep!<P>

Mark Johnson<P>

<STRONG> What about Sun's new printing features?</STRONG><P><P>
Jean-Pierre,<P>

This tutorial is useful and appropriate in presentation. But Sun has introduced new features to enhance Java's printing programmability. The article should also focus on issues like enumerating the available printer's name, selecting a tray, and printing to a network printer using a URL without the printing dialog.<P>

Hasan<P>

<FONT COLOR="#006600"><P>

Hasan,<P>

First, almost every feature you mentioned will be part of the next version of Java; Sun will add the <code>javax.print</code> package, which will support printer discovery, stream printer discovery, translation of 2D graphics to printer language, etc. Some new printing features were added to JDK 1.3, but are not worth mentioning because they were served to us as "side orders." The new <code>JobAttributes</code> class can only be linked to a <code>PrintJob</code> object -- not to a <code>PrinterJob</code> object. If you try to print with a <code>PrintJob</code> object, you will lose all of the functionality provided by the <code>Pageable</code> (<code>Books</code>) printing model. <P>

My second point concerns the objective of this series. My main goal is to provide the reader with a framework that will simplify the creation of printed output in Java. All of the new features you cited are irrelevant at this time, since they can be easily integrated into the framework when Sun releases JDK 1.4.  However, Java programmers need to know how to generate contents; that's what this series focuses on.<P>

Jean-Pierre<P>

<strong> Legacy Systems </strong><br></font><P>
<strong><a href="/javaworld/jw-10-2000/jw-1006-legacy.html"><font size="+1">'Leverage legacy systems with a blend of XML, XSL, and Java'</font></a></strong><br><P>
<em> Michael Koch </em><br><P>

<STRONG> Using transmission-method APIs with mainframe applications</STRONG><P><P>
Mr. Koch,<P>
You detailed a wonderful transaction framework for accessing data via mainframe applications.  However, you purposefully did not delve into any transmission methods that could invoke the mainframe application. I am researching full solutions for interfacing 3270 mainframe applications with Java, and I hope you can comment on your experience with possible transmission methods.  More specifically, I think the OHIO API, drafted by the Internet Engineering Task Force, would fit nicely in your transaction framework. I found three vendors that provide an implementation of the OHIO API:<P>

<ul><P>
<li>Eicon Technology: Aviva for Java</li><P>
<li>Platypus Partners: Jet API</li><P>
<li>Zephyr: Eclipse Host Integration Server</li><P>
 </ul><P>

I also discovered that IBM developed a seeming competitor to the OHIO standard with its HACL (host access class libraries). <P>

Do you have any experience with these transmission-method APIs?  If not, could you detail the transmission methods you have used in the past? <P>

Eric Simmerman<P>

<FONT COLOR="#006600"> <P>

Mr. Simmerman,<P>

Thank you for your comments on my article.  Since I have used many different transmission methods, and so many are available, I did not want to pick one and I omitted that topic from my article.<P>

I have frequently used an IBM product called MQSeries. You can install MQ on a wide variety of legacy systems, and it has many benefits.<P>

MQSeries has a Java API (among others) that ties into the server.  I completed most of my work on an NT platform, but Unix versions do exist -- the API is identical.  MQSeries can also be configured to eliminate much of the grunt work that would typically be involved, such as character-set conversions and numeric conversions.  Simply configure your queues, push in the data, and voila. <P>

Basically, install an MQSeries server on the host and legacy system, and set up the servers to talk to each other. The legacy system can retrieve messages using products that act as listeners on the queues. They pick up and dispatch messages to the appropriate region of the mainframe, and return the results to the other system.  You can write code to do that; it is not difficult.<P>

Visit the <a href="http://www.ibm.com/">IBM Website</a> for more information on MQSeries. It is not expensive -- not cheap, either -- if cost is a consideration.<P>

In another case, I had to talk to a proprietary HP3000.  The only way to penetrate it and execute transactions from the outside was with a Web server application built specifically for that system. It used CGI scripts that could execute the COBOL transactions. In that case, I used the same framework and Java sockets, emulated a Web browser by sending HTTP requests to fire off the CGI scripts, and passed the data as post parameters.  That also worked pretty well.<P>

I am somewhat familiar with CICS Gateway for Java (another IBM product), although I have never used it.<P>

In some cases, I was fortunate that the underlying database was relational and could use JDBC.  That didn't stop me from using the framework; I just created an SQL markup language in XML and completed it that way.<P>

I hope this helps.<P>

Mike Koch<P>

<em>Todd Sunsted</em><br><P>

<STRONG> Does transmitting objects through javax.crypto affect performance?</STRONG><P><P>
Todd,<P>

How does the performance of transmitting objects via the <code>javax.crypto</code> API differ from, say, HTTPS?<P>

When using an object as a parameter in an EJB remote interface method call, would it be faster to use a <code>Sealed</code> object? <P>

Also, do any security compromises quickly come to mind?<P>

Alex<P>
<FONT COLOR="#006600"><P>

Alex,<P>

You asked good questions.  <P>

Here's how I'd make the decision: HTTPS encrypts everything that passes over the wire.  This may not be necessary; depending on the application, it may be too expensive.  On the other hand, if you have Sun's SSL libraries installed, creating an HTTPS connection is as easy as creating a URL.  <P>

So, you should decide whether all of the data needs to be encrypted, whether the application can afford to encrypt it, and whether ease of use is a factor.<P>

Todd<P>

<STRONG> Errors in file naming </STRONG><P><P>
Humphrey,<P>
