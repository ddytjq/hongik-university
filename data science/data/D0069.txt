<h1 align="CENTER">Device programming with MIDP, Part 1 </h1>
<h3 align="CENTER"> The concepts behind MIDP APIs and J2ME to build cross-<em>wireless</em>-platform 
  apps </h3>
<blockquote> <strong>Summary</strong><br>
  Without too much platform customization or rework, vendors can use MIDP-compliant 
  devices to develop applications that can run on multiple wireless platforms. 
  This article, the first in a three-part series, introduces you to the concept 
  of MIDP APIs and the J2ME platform. You will be exposed to the APIs used to 
  generate graphical, form-based, storage-driven code that is capable of connecting 
  with external resources. (<em>3,500 words</em>) </blockquote>
<p><strong>By <a  class=linkColor  href="mailto:michael.cymerman@javaworld.com">Michael Cymerman</a></strong> 
</p>
<p>If you've been following the wireless industry, you've noticed the speed at 
  which a technology runs through its life cycle. A brand new product can become 
  obsolete within months due to an improved technology or platform. The result 
  of that rapid change has been the lack of standardization between the deployment 
  platforms in the wireless world. A development team currently needs its staff 
  to understand numerous markup languages and processes in order to deploy a product 
  onto multiple devices. A marketing department continually needs to modify product 
  direction based on the changing marketplace. </p>
<p> Devices that are compliant with Mobile Information Device Profile (MIDP) will 
  enable vendors to develop applications that can run on multiple wireless platforms 
  without spending intensive amounts of energy customizing or reworking each platform. 
  That would, in turn, let developers focus their energies on the system's functionality. 
<p> This article is the first in a three-part series designed to introduce you 
  to the concept of MIDP APIs and the Java 2 Micro Edition (J2ME) platform. I 
  will expose you to the APIs used to generate graphical, form-based, storage-driven 
  code that can connect with external resources. 
<p> <font size="+1"><strong>Introduction to J2ME </strong></font><br>
  The J2ME is Sun Microsystems's attempt to port the Java programming language 
  to devices with resource limitations. A mobile phone, which lacks the computational 
  power, memory, and workstation power, cannot perform the same functionality 
  as high-end servers or client workstations. 
<p> The J2ME platform is built upon the Java programming language to provide the 
  maximum functionality available on the resource-limited device. A subset of 
  the base functionality is provided along with some specialized classes. 
<p> In this article, I will focus on the CLDC (Connected Limited Device Configuration) 
  and MIDP classes. Those sets of classes make up a <em>profile</em> in the J2ME 
  terminology. That profile is based on the extremely limited device memory, processor 
  speed, battery, and network connectivity bandwidth. 
<p> <font size="+1"><strong>Introduction to CLDC APIs </strong></font><br>
  The CLDC is the base platform on which the MIDP APIs are stacked. The CLDC classes 
  consist of a standardized set of functionality that all vendors who offer J2ME-certified 
  phones will support. Generally, you won't have to interact directly with those 
  classes, but certain devices require that you access those lower-level classes 
  to perform certain functionality. Those low-level accesses will likely be deprecated 
  as the devices and platform develop. 
<p> <font size="+1"><strong>Introduction to the MIDP profile </strong></font><br>
  The MIDP profile has been developed to support the vertical niche of cell phones 
  or similar devices constrained by screen and keypad limitations, in addition 
  to the obvious battery, processor, and bandwidth constraints. 
<p> That profile contains a series of APIs that let you create anything from video 
  games using customized graphics to full-scale business applications using external 
  and internal data sources. 
<p> Several device manufacturers have already announced that they will support 
  the MIDP platform on their devices. At the time of this writing, Nextel has 
  announced that its phones will support MIDP in the first quarter of 2001. 
<p> <font size="+1"><strong>Quick example </strong></font><br>
  Below is a simple HelloWorld type example that offers a quick overview of a 
  midlet's life cycle. A midlet is the name given for the code that executes on 
  your mobile device. It is similar to an applet in that it contains user interface, 
  data, and control capabilities. 
<p> <code> import javax.microedition.midlet.*; <br>
  import javax.microedition.lcdui.*; <br>
  <br>
  public class HelloMidlet extends MIDlet implements CommandListener <br>
  { <br>
  &nbsp;&nbsp;&nbsp;&nbsp;// Initialize the Midlet Display variable <br>
  &nbsp;&nbsp;&nbsp;&nbsp;private Display midletDisplay; <br>
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;// Initialize a variable for the doneCommand <br>
  &nbsp;&nbsp;&nbsp;&nbsp;private Command doneCommand; <br>
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;public HelloMidlet() <br>
  &nbsp;&nbsp;&nbsp;&nbsp;{ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;// Retrieve the display from the static 
  display object <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;midletDisplay = Display.getDisplay(this); 
  <br>
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;// Initialize the doneCommand 
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;doneCommand = new Command("DONE", Command.SCREEN, 
  1); <br>
  &nbsp;&nbsp;&nbsp;&nbsp;} <br>
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;/** <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;* Create the Hello Midlet World TextBox and associate 
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;* the exit command and listener. <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;*/ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;public void startApp() <br>
  &nbsp;&nbsp;&nbsp;&nbsp;{ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;// Create the TextBox containing 
  the "Hello Midlet World!!" message <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;TextBox textBox = new TextBox("Hello 
  Midlet", "Hello Midlet World!!", 256, 0); <br>
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;// Add the done Command to the 
  TextBox <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;textBox.addCommand(doneCommand); 
  <br>
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;// Set the command listener 
  for the textbox to the current midlet <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;textBox.setCommandListener( 
  (CommandListener) this); <br>
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;// Set the current display of 
  the midlet to the textBox screen <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;midletDisplay.setCurrent(textBox); 
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;} <br>
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;/** <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;* PauseApp is used to suspend background activities 
  and release <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;* resources on the device when the midlet is not 
  active. <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;*/ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;public void pauseApp() <br>
  &nbsp;&nbsp;&nbsp;&nbsp;{ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;} <br>
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;/** <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;* DestroyApp is used to stop background activities 
  and release <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;* resources on the device when the midlet is at 
  the end of its <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;* life cycle. <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;*/ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;public void destroyApp(boolean unconditional) <br>
  &nbsp;&nbsp;&nbsp;&nbsp;{ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;} <br>
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;/* <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;* The commandAction method is implemented by this 
  midlet to <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;* satisfy the CommandListener interface and handle 
  the done action. <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;*/ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;public void commandAction(Command command, Displayable 
  screen) <br>
  &nbsp;&nbsp;&nbsp;&nbsp;{ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;// If the command is the doneCommand <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;if (command == doneCommand) <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;{ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;// Call 
  the destroyApp method <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;destroyApp(false); 
  <br>
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;// Notify 
  the midlet platform that the midlet has completed <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;notifyDestroyed(); 
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;} <br>
  &nbsp;&nbsp;&nbsp;&nbsp;} <br>
  } <br>
  </code>

  J2ME Windows Toolkit <code>DefaultGrayPhone</code> emulator.

<p> The first two lines import midlet-specific classes to support the <code>MIDlet</code> 
  class, the <code>CommandAction</code> interface, and the user interface (UI) 
  classes. Those classes are present in the MIDP APIs along with a modified subset 
  of the Java programming language. The classes included in the MIDP API will 
  be discussed in more detail later in this article. 
<p> <code> public class HelloMidlet extends MIDlet implements CommandListener 
  <br>
  </code> 
<p> This line declares the <code>HelloMidlet</code> class to extend the <code>MIDlet</code> 
  class and implement the <code>CommandListener</code> interface. Each midlet 
  must extend the abstract <code>MIDlet</code> class, which contains three methods, 
  described below, that each midlet must override to complete its life cycle. 
<p> 
<table width="85%" align="center" border="1" cellpadding="1" cellspacing="1" bgcolor="#ffffcc">
  <tr align="left"> 
    <th>Method Name</th>
    <th>Method Purpose</th>
  </tr>
  <tr align="left"> 
    <td><code>startApp</code></td>
    <td>To allocate desired system resources and initialize the application.</td>
  </tr>
  <tr align="left"> 
    <td><code>pauseApp</code></td>
    <td>To temporarily suspend resource-intensive processes.</td>
  </tr>
  <tr align="left"> 
    <td><code>destroyApp</code></td>
    <td>To release resources used by the midlet and dispose of the midlet.</td>
  </tr>
</table>
<p> In the previous example, the <code>startApp</code> contains the bulk of the 
  functionality because the <code>HelloMidlet</code> does not use other system 
  resources such as network connections or datastores. When the <code>HelloMidlet</code> 
  is executed, the <code>startApp</code> will be called. 
<p> As you would expect, the constructor is executed prior to the <code>startApp</code> 
  method. In this example, the constructor retrieves the display from the global 
  <code>Display</code> object. In addition, the constructor initializes the <code>doneCommand</code>. 
<p> The <code>startApp</code> method above is used to create the screen to be 
  displayed on the midlet. The following line shows the initialization of the 
  <code>TextBox</code> screen. 
<p> <code> TextBox textBox = new TextBox("Hello Midlet", "Hello Midlet World!!", 
  256, 0); <br>
  </code> 
<p> A midlet <code>Screen</code> may also contain <code>Commands</code>. A command 
  is the mechanism the midlet uses to create menus as shown in the picture above 
  with the "Done" button. Here's the code used to add the "Done" button to the 
  <code>TextBox</code> screen: 
<p> <code> &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;// Add the done Command to the 
  TextBox <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;textBox.addCommand(doneCommand); <br>
  </code> 
<p> The <code>Command</code> source will generate <code>CommandActions </code> 
  when it is clicked. In this simple example, the <code>HelloMidlet</code> will 
  implement the <code>CommandListener</code> interface and thereby handle all 
  events itself. As the developed system becomes more complex, handling events 
  in a more expanded manner may be beneficial. 
<p> <code> &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;// Set the command 
  listener for the textbox to the current midlet <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;textBox.setCommandListener( 
  (CommandListener) this); <br>
  </code> 
<p> Now that the <code>TextBox </code> has been created, it is ready to be displayed 
  on the device screen. In the following line, I am setting the current display

<p> <code> &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;// Set the current display of the 
  midlet to the textBox screen <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;midletDisplay.setCurrent(textBox); <br>
  </code>

<p> <strong>Building the midlet </strong><br>
  The J2MEWTK contains a built-in build tool that will perform the build process 
  for your project. To build your project, click on the Build button shown in 
  Figure 2. That toolkit contains an automated build process, but others may not. 
  Therefore, here are the steps of the build process: 
<p> 
<ol>
  <li>Create the classes and tmpclasses directories for build processing</li>
  <li>Compile the Java source files into the tmpclasses directory</li>
  <li>Preverify the classfiles contained in tmpclasses and output to classes</li>
  <li>Jar up the verified classfiles</li>
  <li>Jar up the resource files</li>
  <li>Create a reminder to update the jar file size in the jad file</li>
</ol>
<p> As a result of that build process, a jar file will contain all the preverified 
  classes needed to deploy that application to the device and the project's jad 
  file. 
<p> <strong>The jad file </strong><br>
  The jad file contains all the information about the midlet, its property names, 
  and its associated properties' values. For the <code>HelloMidlet</code> class 
  used in the example, the jad file is: 
<p> <code> MIDlet-1: HelloMidlet, HelloMidlet.png, HelloMidlet <br>
  MIDlet-Jar-Size: 1387 <br>
  MIDlet-Jar-URL: HelloMidlet.jar <br>
  MIDlet-Name: HelloMidlet <br>
  MIDlet-Vendor: Sun Microsystems <br>
  MIDlet-Version: 1.0 <br>
  </code> 
<p> The midlet name, URL, size, and version are among the entries listed. The 
  most interesting entry may be the <code>MIDlet-1</code> line above. There is 
  only one midlet in this jad file and, therefore, only one line. If there were 
  n midlets, there would be n lines as follows: 
<p> <code> MIDlet-1: ... <br>
  MIDlet-2: ... <br>
  ... <br>
  MIDlet-n: ... <br>
  </code> 
<p> Each of the above lines would provide the midlet's name, the midlet's image 
  file, and the name of the class file containing the midlet. 
<p> <strong>Why pre-verify? </strong><br>
  As mentioned above, the class files are run through the preverification process. 
  That process ensures that the classfiles do not attempt any of the invalid operations 
  described previously. In addition, it is beneficial to perform some steps of 
  the standard virtual machine on a more capable computer rather than leave them 
  to the device's KVM. 
<p> The KVM, designed to fit within 1 K, has been optimized for processing on 
  the devices, not basic duties that you can outsource through preverification.

<p> <font size="+1"><strong>Graphical framework example </strong></font><br>
  The following example constructs a simple graphical framework to enable you 
  to efficiently develop graphics code within the MIDP environment. 
<p> A simple graphical example will include the following classes: 
<p> 
<ul>
  <li><code>GraphicalMidlet</code>: The midlet class handling interactions with 
    the device</li>
  <li><code>GraphicalCanvas</code>: The canvas containing the graphical objects</li>
  <li><code>GraphicalObject</code>: A default graphical object that implements 
    <code>Runnable</code></li>
  <li><code>MovingTextObject</code>: The default moving text object</li>
  <li><code>VerticalScrollingTextObject</code>: An extension to the default text 
    object</li>
</ul>
<p> In that example, the <code>GraphicalMidlet</code> class is responsible for 
  extending the <code>MIDlet</code> class and thereby enforcing the midlet's life 
  cycle. In addition, that class is responsible for implementing the <code>CommandAction</code> 
  interface to handle the single command of Exit when the user is ready to leave 
  the application. 
<p> The constructor of the <code>GraphicalMidlet</code> class retrieves the display 
  object from the statically accessible <code>Display</code> object. The constructor 
  then instantiates a <code>GraphicalCanvas</code> class for which it will perform 
  as the <code>CommandListener</code>. 
<p> <code> &nbsp;&nbsp;&nbsp;&nbsp;public GraphicalMidlet() <br>
  &nbsp;&nbsp;&nbsp;&nbsp;{ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;display 
  = Display.getDisplay(this); <br>
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;canvas 
  = new GraphicalCanvas(display); <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;canvas.addCommand(exitCommand); 
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;canvas.setCommandListener(this); 
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;} <br>
  </code> 
<p> In the <code>GraphicalCanvas</code> constructor, a new <code>VerticalScrollingTextObject</code> 
  is created passing the canvas and the message. In this simple framework, no 
  additional information regarding the speed, positioning, or the text message 
  color is included. This expansion is left to the reader as an exercise. 
<p> <code> &nbsp;&nbsp;&nbsp;&nbsp;public GraphicalCanvas(Display d) <br>
  &nbsp;&nbsp;&nbsp;&nbsp;{ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;display = d; // save the display <br>
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;graphicalObject = <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;new VerticalScrollingTextObject 
  (this, "Hello Midlet World!!"); <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;height = this.getHeight(); <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;width = this.getWidth(); <br>
  &nbsp;&nbsp;&nbsp;&nbsp;} <br>
  </code> 
<p> The <code>paint</code> method is used by the <code>GraphicalCanvas</code> 
  to redraw the screen as needed by the application. In this case, the canvas 
  clears the previous entry and then requests that the <code>GraphicalObject</code> 
  paint itself. 
<p> <code> &nbsp;&nbsp;&nbsp;&nbsp;protected void paint(Graphics g) <br>
  &nbsp;&nbsp;&nbsp;&nbsp;{ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;g.setColor(255,255,255); 
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;g.fillRect(0,0,width,height); 
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;graphicalObject.paint(g); 
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;} <br>
  <br>
  </code> 
<p> The <code>start()</code> method is invoked by the <code>GraphicalMidlet</code> 
  to alert the <code>GraphicalCanvas</code> to begin its action. In this example, 
  the <code>canvas</code> class instantiates a new thread containing the <code>Runnable 
  GraphicalObject</code> class. The <code>canvas</code> class then starts the 
  new thread so that it may begin its processing prior to calling the <code>repaint</code> 
  method to initialize the painting of the canvas. 
<p> You can also expand this example by adding more than one <code>GraphicalObject</code> 
  to the canvas and then executing them simultaneously. I will leave that for 
  you as a simple exercise. 
<p> <code> &nbsp;&nbsp;&nbsp;&nbsp;void start() <br>
  &nbsp;&nbsp;&nbsp;&nbsp;{ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;display.setCurrent(this); 
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;Thread 
  t = new Thread(graphicalObject); <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;t.start(); 
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;repaint(); 
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;} <br>
  </code> 
<p> The <code>GraphicalObject</code> class implements the <code>Runnable</code> 
  interface and can therefore be started by the thread in the <code>GraphicalCanvas</code> 
  class's <code>start()</code> method. This <code>Thread</code> will execute until 
  the <code>stopThread</code> method is called to break out of the loop. 
<p> <code> &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;public void run() <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;{ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;stopThread 
  = false; <br>
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;while 
  (!stopThread) <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;{ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;move(); 
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;canvas.repaint(); 
  <br>
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;try{ 
  Thread.sleep(100); <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;}catch(InterruptedException 
  ie){ stopThread = true;} <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;} <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;} <br>
  </code> 
<p> The <code>GraphicalObject</code> does not know the desired movement and painting 
  of its subclass objects, but it specifies two methods that must be overridden. 
  They are as follows: 
<p> <code> &nbsp;&nbsp;&nbsp;&nbsp;public abstract void move(); <br>
  &nbsp;&nbsp;&nbsp;&nbsp;public abstract void paint(Graphics g); <br>
  </code> 
<p> By overriding the methods above, the <code>MovingTextObject</code> can implement 
  the movement it desires and the painting scheme as well. So, to generate a different 
  object, such as a rotating gear, you would just have to change the <code>paint</code> 
  and <code>move</code> methods. 
<p> <code> &nbsp;&nbsp;&nbsp;&nbsp;public void paint(Graphics graphics) <br>
  &nbsp;&nbsp;&nbsp;&nbsp;{ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;graphics.setColor(0,0,0); 
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;graphics.drawString(stringValue, 
  posX, posY, Graphics.TOP| Graphics.LEFT); <br>
  &nbsp;&nbsp;&nbsp;&nbsp;} <br>
  <br>
  &nbsp;&nbsp;&nbsp;&nbsp;public void move() <br>
  &nbsp;&nbsp;&nbsp;&nbsp;{ <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;posX = getX(); <br>
  &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;posY = getY(); <br>
  &nbsp;&nbsp;&nbsp;&nbsp;} <br>
  </code> 
<p> In the <code>move()</code> method above, the methods <code>getX()</code> and 
  <code>getY()</code> are called to return the moving text object's X and Y position. 
  I did that to demonstrate an expansion to the framework. In this case, the expansion 
  is fairly simple, the <code>getX()</code> method returns a constant value, while 
  the <code>getY()</code> method increments the position of the Y value until 
  the bottom of the display is reached. 
<p> You could add additional movement schemes by subclassing the <code>MovingTextObject</code>. 
  For example, a <code>RandomMovingTextObject</code> class might implement random 
  values for the X and Y location points and return those in the <code>getX</code> 
  and <code>getY</code> methods. I'll leave further expansion to the reader. 
<p> <font size="+1"><strong>What's ahead in this series </strong></font><br>
  My intent for this first article was to introduce the MIDP concept, demonstrate 
  the life cycle through a simple code example, and illustrate a graphical example 
  similar to the basic applets that most of you have constructed at one point 
  or another. 
<p> In Part 2 in the series, I will discuss the rudimentary storage system that 
  the MIDP provides on the wireless device. Using form-based examples, I will 
  analyze the APIs that you can use to provide a wireless database. 
<p> In Part 3, I will detail the APIs used to connect the wireless device to external 
  data sources such as URLs. In addition, a more detailed application will illustrate 
  some concepts that you can follow to make MIDP development more efficient. 
<p> <font size="+1"><strong>Conclusion </strong></font><br>
  The deployment of Java code to the wireless device is the next logical step 
  in the progression of functionality to that platform. The midlet has the potential 
  to revolutionize the wireless platform by letting developers target mobile users 
  more directly with more functional, easier-to-use applications. When combined 
  with some of the other enhancements in the industry, such as location sensing, 
  the platform appears poised for immediate acceptance. 
<p> As was mentioned earlier, the MIDP platform will be present on devices beginning 
  in January 2001. Nextel has planned support for the distribution of Java-enabled 
  Motorola phones starting in January 2001. 
<p> This is an exciting area of development that lets developers enter the wireless 
  space without learning too many new Java APIs. The platform gives the "Write 
  Once, Run Anywhere" concept in Java a whole different meaning. <img height="8" width="8" src="http://a301.g.akamai.net/7/301/1403/v001/www.javaworld.com/javaworld/icons/dingbat.gif">
