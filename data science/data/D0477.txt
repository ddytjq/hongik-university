    They could at least bother to support the de facto standard for Java GUIs, like it or not it isn't going away anytime soon. As long as it doesn't support swing I have no use for it. I've only played with eclipse for a short moment. It looks pretty decent but I'd say swing looks prettier. I wasn't impressed with the speed either. SWT or not, it loads about as fast as netbeans on my machine. That's not so strange since both actually spend most of their initialization time initializing plugins. <BR><BR>Most proponents of SWT cite either performance or look and feel as the advantages. However, on my 2 1/2 year old PC swing performance is very satisfactory with jre 1.4. I'm pretty happy running  Jedit and limewire for instance, both of which work at near native speeds on my machine (500 Mhz pIII, 512MB). And as has been pointed out, creating a SWT look and feel for swing is easy (not that I'd use it anyway).<BR><BR>BTW. how does one distribute SWT applications? I haven't actually seen any yet (except for the IDE) and I suppose you need SWT preinstalled in order to run SWT apps. Since SWT requires native components stuff like webstart is probably not an option.<BR>

    I just read through everything I could find about SWT again.  I would understand if they made SWT for the tool itself (sort of like the old IDE's that were mostly native code) but it sounds like they really want to make it a toolkit for everyone to use.<BR><BR>They say a wise developer would have to test their application on every target platform... the same is true of Swing applications but in a much different sense.  With SWT you'd be testing behavior of native components for things like automatically selected radio buttons on windows but no on the mac, window focusing, etc.  With Swing you'd be testing to see if you hit any of the Swing bugs on the various platforms. I guess that's ok but it means that the application won't be as widely accepted as it would without the libraries.<BR><BR>I just checked the some sizes:<BR>swt.jar 971 KB<BR>swt-win32x.dll 256 KB<BR><BR>Not bad.<BR><BR>If parts of the JVM were loaded already, Swing apps would start very very fast.  To test this, try writing a custom classloader for an application only, not the core classes and then run the app.  The difference is more than dramatic.  If Sun were to push a shareable VM through, startup issues wouldn't be a problem.  Runtime speed would still be a problem.<BR><BR>I'm all for a faster crossplatform GUI.  I don't care so much about native platform L&F but it certainly should be available.  It seems that Swing needs to be more flexible or we need better L&F's to handle the native look and feels.  Apple has managed to do it better than any other L&F.  It should be possible for windows as well.<BR><BR>I guess my ultimate concern is that SWT doesn't seem to fit in the WORA mentality.  Hopefully if SWT becomes popular, it will exist on as many platforms as possible instead of a just 1.5 (Windows and sort linux and solaris).  I just noticed that late last month they had a GTK1.2 version available.  I'll take a look at it.<BR><BR>Another thing that concerns me is that SWT is more conducive to breaking when the native UI system changes.  This is true of AWT/Swing but not as much so as SWT.  Hopefully SWT would be released simultaneously with the new platform... ie when GTK2 comes out, I hope Eclipse is immediately available.<BR>
