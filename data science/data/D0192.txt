<STRONG>Summary</STRONG><BR>

Last month's <strong>JavaBeans</strong> column introduced IBM's XML Beans, a suite of JavaBeans that process XML files. This month, Mark looks at the <code>XMLEditor</code> and <code>XMLConvenience</code> bean sets that make creating custom XML editors a snap. <em>(3,400 words)</em>

Last month's components showed you how to read and write XML trees. This month, I'll show you how to use the nonvisual <code>XMLEditor</code> bean set to process those trees. This article will be highly technical, so you must have prior knowledge of how JavaBeans interconnect in an integrated development environment (IDE) at the lowest level. I'll give a detailed example of a very simple application, with explanations of event and property connections, and describe how you can use them to create applications <em>without</em> programming. <P>
This article assumes you've read Part 1 of this series and that you're familiar with the basics of XML, the Document Object Model (DOM), and JavaBeans, as well as the concept of building applications in an IDE. It would also be helpful to download and install IBM's XML Beans from IBM's alphaBeans site (see <a href="#resources">Resources</a>). <P><FONT SIZE="+1"><STRONG> The XMLEditor bean set </STRONG></FONT><BR>
<code>XMLEditor</code> beans do not necessarily edit XML, despite what their name implies. Instead, these beans form a hierarchy of containers and operators that match particular features of an <code>org.w3c.dom.Document</code> or a <code>org.w3c.dom.Node</code> (which I will simply call <code>Document</code>, <code>Node</code>, <code>Element</code>, etc.), and hold a reference to that object for processing. <P>
That object represents a part of the input XML tree being edited. As shown last month, a <code>DOMGenerator</code> bean creates a <code>Document</code> object, representing the tree of XML nodes. But that capability is almost useless if you can't pick the tree apart and use its pieces. That's where <code>XMLEditor</code> beans come in. <P>
I'll start with an example that clarifies what I'm discussing here. Part of the sample XML I've been using for these articles, a recipe for Lime Jello Marshmallow Cottage Cheese Surprise, appears in Figure 1 in two different forms. <P>

The left side of the figure represents part of the sample XML document, encoded as ASCII text. (You can find a link to the entire sample XML code in the <a href="#resources">Resources</a> section below.) On the right is the same document, represented as a tree of DOM <code>Node</code> objects (subclasses of <code>org.w3c.dom.Node</code>). The <code>DOMGenerator</code> (by way of its underlying parser) produces this tree from the input XML stream. <P>
The top-level element is the <code>Document</code> node, only one of which exists for any given XML document. The <code>DOMGenerator</code> object creates an object of this type and provides it as its <code>result</code> property. A <code>Document</code> is a container that, if not empty, holds one and only one object of type <code>Element</code>, which is in turn given the same name as the <code>Document</code>'s top-level tag. This element is called the <em>document element.</em> In this example, the document element is <code>&lt;Recipe&gt;</code>. (With the notation <code>&lt;<i>Tagname</i>&gt;</code>, I indicate both the name of an XML tag and the tag name returned by the <code>getTagName()</code> method of a corresponding DOM <code>Element</code> object.) <P>
The <code>&lt;Recipe&gt;</code> contains a single <code>&lt;Name&gt;</code> that contains a <code>Text</code> item. The <code>Text</code> item's value is the name of the recipe. This structure is defined and enforced by the Document Type Definition (DTD) for the XML document. (You can see the simple DTD for this Recipe XML in <a href="#resources">Resources</a> below.) <P>
What does all this have to do with <code>XMLEditor</code> beans? Each container bean class in the <code>XMLEditor</code> bean set recognizes a particular <code>Node</code> type at the bean class's input. For example, a <code>DocumentContainer</code> <code>XMLEditor</code> bean takes a <code>Document</code> node as its input. A <code>DocumentContainer</code>'s input is the value of its <code>inputDocument</code> property, which is write only. The <code>DocumentContainer</code> contains a read-write property called <code>currentNode</code>, which is the document to be edited. The <code>DocumentContainer</code> has a third property, <code>result</code>, which is read only, and returns the <code>Document</code> object that the <code>DocumentContainer</code> currently contains. <P>
The <code>inputDocument</code> (write only) and <code>result</code> properties (read only) can be different because the document within the <code>DocumentContainer</code> (the <code>currentNode</code>) is editable. After editing (setting the <code>currentNode</code> property to refer to a different <code>Document</code> object, for example), the <code>result</code> document -- that is, the output -- may well be different from the input.
<P>
Let's apply this to the example above. Imagine setting a <code>DocumentContainer</code>'s <code>inputDocument</code> to the <code>Document</code> object at the top of the object tree in Figure 1. The <code>DocumentContainer</code> will make the <code>Document</code> available as a <code>Node</code> (since a <code>Document</code> is a subclass of <code>Node</code>, and therefore <em>is</em> a <code>Node</code>), and then make that <code>Node</code> available as its <code>currentNode</code> property. It will also make the <code>Document</code> available as its <code>result</code> property. <P>
To understand how an application composed of interconnected objects works, you must understand how to interconnect objects. What does it mean for objects to be connected to one another? <P>

Returning to the example in Figure 1, let's build a small application that gives the user a view of a recipe's name -- that is, the contents of the <code>&lt;Name&gt;</code> element inside a <code>Document</code>'s <code>&lt;Recipe&gt;</code> element. Figure 2 shows the configuration of just such an application running inside the IBM VisualAge for Java IDE. Other IDEs may present the connections between beans in completely different ways, but all serious IDEs allow connections between bean properties, events, and methods. <P>

The purpose of the <code>RecipeEditor</code> bean is simply to extract the text of the <code>&lt;Name&gt;</code> element of a <code>Document</code> containing a <code>&lt;Recipe&gt;</code> and display that <code>&lt;Name&gt;</code> to the user. <P>
For the rest of this section, I'm going to delve into the inner workings of how IDEs connect objects to one another, using the simple <code>RecipeEditor</code> as an example. If you already understand event listener relationships and the ways in which IDEs interconnect components to create applications, you can skip down to the section titled <A HREF="#unprogrammed"> <strong>An unprogrammed application</strong> </a>. <P>
Clicking the Open File button causes the <code>DOMGenerator</code> object (marked A in Figure 2) to read its input, parse any XML it finds, and produce a DOM <code>Document</code> as its output. But how exactly does clicking the button cause this to happen? <P><STRONG> Making the connection </STRONG><BR>

The arrow in Figure 2 between the Open File button and <code>DOMGenerator1</code> indicates an event listener relationship. In this case, the <code>DOMGenerator</code>'s <code>triggerAction()</code> method is called when the Open File button produces an <code>ActionEvent</code>. Calling <code>triggerAction()</code> causes the <code>DOMGenerator</code> to fetch its input document, parse the document, and produce a <code>Document</code> object as its result. <P>
It's fairly easy to create a <code>DOMGenerator</code> that listens for <code>ActionEvent</code>s on another object. For example: <P>
<CODE><P>
<BR>

public class DOMGeneratorThatListens<BR>

&nbsp;&nbsp;extends DOMGenerator implements ActionListener {<BR>

&nbsp;&nbsp;&nbsp;&nbsp;public void actionPerformed(ActionEvent e) {<BR>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;triggerAction();&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; <BR>

&nbsp;&nbsp;&nbsp;&nbsp;}<BR>

}<BR>
</CODE><P>
Then, elsewhere in the code: <P>
<CODE><P>
<BR>

Button openFile = new Button("Open file...");<BR>

DOMGeneratorThatListens dg = new DOMGeneratorThatListens();<BR>

openFile.addActionListener(dg);<BR>
</CODE><P>
Simple enough. In real applications, though, many objects listen for events from multiple sources, and they may also send events to other objects. The resulting code quickly becomes too complex for an IDE to handle easily. <P><STRONG> Being adaptable </STRONG><BR>
A common solution to this problem is to split the listening and reacting functionality out of the subclass and place it in a separate class, called an <em>event adapter</em> class. An event adapter is simply a dispatcher that listens for events on an event source and then calls methods on the event target. Code for an event adapter might look like this: <P>
<CODE><P>
<BR>

public class DOMGeneratorButtonAdapter implements ActionListener {<BR>

&nbsp;&nbsp; DOMGenerator dg = null;<BR>

&nbsp;&nbsp; Button b = null;<BR>

&nbsp;&nbsp; DOMGeneratorButtonAdapter(Button source, DOMGenerator target) {<BR>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;b = source;<BR>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;dg = target;<BR>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;b.addPropertyChangeListener(this);<BR>

&nbsp;&nbsp; }<BR>

&nbsp;&nbsp; public void actionPerformed(ActionEvent e) {<BR>

&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;dg.triggerAction();<BR>

&nbsp;&nbsp; }<BR>

}<BR>
</CODE><P>
You could then link together a button and a <code>DOMGenerator</code> <em>without</em> subclassing either of them, like this: <P>
<CODE><P>
<BR>

Button openFile = new Button("Open file...");<BR>

DOMGenerator dg = new DOMGenerator();<BR>

DOMGeneratorButtonAdapter dgba = new DOMGeneratorButtonAdapter(openFile, dg);<BR>
</CODE><P>
When the button is clicked, the adapter receives the event and then performs the method call on the target object. <P>
A purely object-oriented solution would be to create a separate event adapter object for each connection. Some IDEs may do so, but the designers of VisualAge for Java decided to turn the application class (in this case, <code>RecipeEditor</code>) into the event adapter for all events. In other words, the application class listens for all events from objects (e.g., clicks on the Open File button) and then dispatches calls to the appropriate targets (e.g., the <code>DOMGenerator</code>'s <code>triggerAction()</code>). <P>
When the user clicks the Open File button, the button sends an <code>ActionEvent</code> to all of its listeners by passing it to those listeners' <code>actionPerformed()</code> methods. The application class (<code>RecipeEditor</code>) is registered as a listener for <code>ActionEvent</code>s, and it responds to the button's <code>ActionEvent</code> by calling the <code>DOMGenerator</code>'s <code>triggerAction()</code> method.
<P><STRONG> How connections are created </STRONG><BR>
In Figure 2, the application developer created the connection between the button and the <code>DOMGenerator</code> in the IDE. The arrow leading from the button to the <code>DOMGenerator</code> is a graphical representation of a method that the IDE generated automatically. The application developer indicated (graphically, using mouse clicks and menus) that the IDE should connect the button's <code>actionPerformed()</code> method to the <code>DOMGenerator</code>'s <code>triggerAction()</code> method. The IDE then automatically took the following steps: <P>
<oL> <LI>It declared the application class <code>RecipeEditor</code> as an <code>ActionListener</code> by adding <code>implements ActionListener</code> to the <code>RecipeEditor</code> class declaration. This made the class capable of receiving <code>ActionEvent</code> objects. <P>
<LI>It wrote a handler method <code>actionPerformed</code>, which detects whether the source of the <code>ActionEvent</code> was the Open File button and, if so, calls <code>triggerAction()</code> on the <code>DOMGenerator</code>. Here's a sample of the code the IDE produced: <P>
<CODE><P>
<BR>

/* WARNING: THIS METHOD WILL BE REGENERATED. */<BR>

public void actionPerformed(ActionEvent e) {<BR>
if ((e.getSource() == getButton3()) ) {<BR>
connEtoM11(e);<BR>
}<BR>
<font color="red">if ((e.getSource() == getOpenFile()) ) {<BR>
connEtoM1(e);<BR>
}</font><BR>

}<BR>

/* WARNING: THIS METHOD WILL BE REGENERATED. */<BR>

private void connEtoM1(ActionEvent arg1) {<BR>
<font color="red">getDOMGenerator().triggerAction();</font><BR>

}<BR>
</CODE><P>
You can see that the IDE wrote two methods: <code>actionPerformed()</code>, which checks the source of the event, and <code>connEtoM1()</code>, which <code>actionPerformed</code> calls after making that check and which actually performs the connection. Notice also that, instead of using the user interface element objects directly, the IDE has created accessors for them (<code>getOpenFile()</code> and <code>getDOMGenerator()</code>). This is good programming style, as it encapsulates access to the class's internal objects. </LI> <P>
<LI>Finally, the IDE registered the application class as a listener for <code>ActionEvent</code>s on the Open File button in the <code>RecipeEditor.initConnections()</code> method, called from the constructor. <code>initConnections()</code> is a method, written entirely by VisualAge for Java, that is responsible for creating property change and event relationships. </LI> </oL> <P>
Code generated by other IDEs would look very different from code that appears in the source files, but it would do essentially the same thing: it would call <code>triggerAction()</code> when the Open File button is pushed. (You can find <code>RecipeEditor.java</code>  in the source archive, downloadable in jar, tgz or zip formats from <a href="#resources">Resources</a> below.) <P>
This is the standard way in which IDEs assemble objects into applications: they create instances of component classes and interconnect them with event listener and property change listener relationships. Each object maintains a list of other objects to notify when particular events occur. When an event occurs on an object (a mouse click, a query result return, a method call, input ready to be read, and so forth), the object generates an event (itself an object) describing what happened, and passes it to all the other objects listening for such events. That way, objects can be assembled into applications without compiling anything at runtime. <P>
A component knows what other objects wish to be notified about an event because, earlier in the program's run, the object was added to the component's list of listeners for that event by way of a call to the component's <code>add<em>eventType</em>Listener()</code> method. <P>
The method call that adds one object to another object's list of listeners can occur practically anywhere in a program. Often components will listen for their own events. Window classes commonly listen for their own <code>WindowEvent</code> messages, for example. Sometimes components will add other objects to their own listener lists. Usually an object adds itself to another object's list of listeners for a particular event, by making a call such as <code>otherObject.addSomeEventListener(this)</code>. In most of the cases in this article,  though, an IDE generates the code that adds objects to other objects' listener lists, creating an event connection between the two objects. VisualAge for Java creates all these connections in a method called <code>initConnections()</code> (which you can see in this month's sample source code; see <a href="#resources">Resources</a> to download it). <P>
The reason I'm spending so much time on this topic is that it's the central concept behind building applications visually. If the ideas are still unclear to you, read up on event listener relationships in <a href="#resources">Resources</a>. Meanwhile, let's return to the sample application. <P>

That's an implementation detail. From the point of view of the person assembling an application, a connection between two components is direct. You need only understand what the IDE is doing, so you can debug more effectively. <P>

<TR> <TD ALIGN="CENTER" BGCOLOR="fff0ff"><strong>G</strong></TD> <TD> The <code>TextContainer</code> is yet another <code>XMLEditor</code> class. <!-- break --><P><FONT SIZE="+1"><STRONG> Why build applications with components? </STRONG></FONT><BR>
This is quite a lot of work and a lot of explanation for something so simple, right?  Seems as though it would be easier just to code the whole mess by hand. But there are advantages to building an application this way. <P>
<ul> <li>First, this application was built with no programming. Not one line. I created a running application that can read an XML file, pick the text out of an element a couple of levels down, and display it for editing in a panel by <em>drawing pictures.</em> And it was built in less time than it took to explain how it works.
<P>
<li>Second, building applications by assembling components lets you create a system at a higher level of abstraction than programming -- in terms of ideas rather than code. To draw an application this way is to work with objects in the domain of the system being built. In this case, the domain is processing XML documents.
<P>
There's still plenty of room for coding -- the components must come from somewhere. But, while you're building a system out of components, the temptation to start coding is an indication that something isn't right with your set of components. Ask yourself why you want to code something manually rather than provide it as a service that a component can offer; your answers will improve your design because you're thinking through that decision consciously. <P>
<li>Third, the designers of this XML JavaBeans package did a good job of separating the functionality of the <code>XMLEditor</code> beans from the user interface. They deliberately gave these beans <em>no</em> user interface, because they'd like them to be used  where a GUI is not appropriate. On a server, for example, decoupling functionality from presentation is one good way to make code reusable.
<P>
It would be easy to disconnect the pieces of the application that have no user interface and use them in another context, completely unmodified. Imagine, for example, that you had a directory with thousands of XML recipe files and you wanted to get their names for an index. Replace the <code>ExtendedFileDialog</code> with a directory-scanning object and send the output of <code>TCName</code> to a self-sorting vector of strings, and you have a program that indexes recipe files. Of course, you may need to write the directory scanner or sort the vector beans yourself, but you'll be able to reuse those in the future if they're general and well designed. <P>
<li>Fourth, there's another form of recycling I haven't explored: using components to build other components. Imagine taking this chain of little objects, putting it in a box, and labeling the box "Recipe Name Finder." The chain of components just created could be used anywhere, whenever the name of a recipe was needed.
<P>
<li>Finally, applications built with components and events can be easier to debug than coded programs, because test classes can be attached to the event connections to report what is going on as the program runs. Debugging a program with events then becomes more like troubleshooting an electrical circuit than debugging code. Again, the person doing the troubleshooting is working at a higher level of abstraction. </ul> <P>
Using components to create applications can cause problems, too. Performance limitations, training considerations, memory usage, and the lack of components that have the exact functionality needed are examples. But that's material for another column. <P><FONT SIZE="+1"><STRONG> Running the sample program </STRONG></FONT><BR>
You need to install the XML Beans suite and include its jar file in your <CODE>CLASSPATH</CODE> to run this month's sample code (see <a href="#resources">Resources</a> below to find it). If you're interested in how the code works, you can load the <code>.java</code> files (also available in <a href="#resources">Resources</a>) into your own IDE and step through the code manually. If you use VisualAge for Java, you should be able to open the <code>RecipeEditor</code> class in the Visual Composition Editor and play with the connections. <P><FONT SIZE="+1"><STRONG> Conclusion </STRONG></FONT><BR>
Of course, the little application you've seen is useless. I doubt anyone's going to be sitting at a desk viewing recipe titles all day. But now that you have the basics down, you can start to do more interesting things with these components. <P>
In the next part of the series, I'll show you how to make the contents of these containers updatable, traverse the DOM tree in more than one direction at once, and handle multiple objects contained in a node. I'll also show how to create such applications more easily by using the <code>XMLConvenience</code> bean set of the XML JavaBeans suite.
