<p align="right"><br>
SOAP - Simple Object Access Protocol - is the hottest thing in XML development
right now. It has a major role to play in Microsoft's next generation of Visual
Studio, and it is the basis of their &quot;.NET&quot; strategy. If you're
looking to write a bullet-proof SOAP service now, in VB6, then look no further
than Microsoft's SOAP toolkit for VB; but if you really want to <b>understand</b> 
what SOAP is all about, then you need to get under the hood, and start building
your own objects around the SOAP standard. This article aims to get you started.</p>
<p align="right">In the article, we will create a simple SOAP service, and a
client to access that service. The service will be in the form of an Active
Server Page, which I have called soap.asp - it will need to live in the root
directory of your Personal Web Service (by default c:\inetpub\wwwroot\) Our
service will accept and process SOAP requests made by our client - a VB exe,
with a single module, and a start-up <i>Sub Main()</i>.</p>
<p class="ChapterTitle" align="right">
<font color="#008080">Simple Object Access Protocol: A Step-By-Step Approach<br>
</font>
<font size="4" color="#000000">By Syd Egan</font>
</p>
<p><img border="0" src="../images/A.gif" align="left">s we know, SOAP is a call-response mechanism, which operates in a
client-server paradigm. The client (your application) makes a <i>call</i> to the
server (a web server, somewhere on the internet), passing in parameters; and the
server provides a <i>response</i>. Both call and response are transported in the
form of XML documents. Therefore, to build our own example SOAP system, we need
both a client and a server - a <i>caller</i> and a <i>responder</i>.</p>
<p>The example we will follow is a very simple one: we will build a service that
will calculate the tax due on a sales transaction. In traditional VB terms well
will create a function with the following definition:</p>
<blockquote>
<p class="code"><font color="#000080">Public Function</font> GetSalesTax(<font color="#000080">ByVal</font>
pSalesTotal <font color="#000080">As Double</font>) <font color="#000080">As
Double</font><br>
&nbsp;&nbsp;&nbsp; GetSalesTax = pSalesTotal * 0.04<br>
<font color="#000080">End Function</font></p>
</blockquote>
<p>Crude, but effective (if you live in a state where the sales tax rate is 4%!)</p>
<p>This function defines a method name (GetSalesTax), a parameter (pSalesTotal -
representing the total value of the sale) and a return value (the result of the
function). In more traditional object-oriented terms, we might think of
pSalesTotal as an &quot;IN&quot; parameter, and the GetSalesTax result as an
&quot;OUT&quot; parameter. Quite straightforwardly: sales-total in; sales-tax
out. A SOAP service can be thought of in terms of &quot;IN&quot; and
&quot;OUT&quot; parameters - the client passes &quot;IN&quot; parameters to the
server, and then receives the &quot;OUT&quot; parameters in return. So, in order
to create our SOAP service, we need a server that will listen for requests to
GetSalesTax, accompanied by &quot;IN&quot; parameters, and which will respond
with &quot;OUT&quot; parameters, indicating the correct tax amount. Let's start
with the client - the <i>caller</i>. How do we call the SOAP service to make our
request.</p>
<p class="ChapterH2">The Client</p>
<p>In traditional VB terms, our request to the GetSalesTax function described
above, would be something such as:</p>
<blockquote>
<p class="code">dblSalesTax = GetSalesTax(100)</p>
</blockquote>
<p>Returning a value of $4.</p>
<p>If the GetSalesTax function was contained within an external object, such as
an MTS server, then the call would need to reference the server DLL:</p>
<blockquote>
<p class="code"><font color="#000080">Dim</font> objTax <font color="#000080">As
New</font> CTaxCalc<br>
dblSalesTax = objTax.GetSalesTax(100)</p>
</blockquote>
<p>In a SOAP system the call is little different, only the request is formatted
as an XML document, which is passed up to the server. The appropriate document
contains the same details as the MTS call -a method to call, and the parameter
name and value.</p>
<blockquote>
<p class="code">&lt;GetSalesTax&gt;<br>
&nbsp;&nbsp;&nbsp; &lt;SalesTotal&gt;100&lt;/SalesTotal&gt;<br>
&lt;GetSalesTax&gt;</p>
</blockquote>
<p>In order to ensure that the server can correctly identify and decrypt this
method call, it is wrapped up in a larger document, called a SOAP envelope,
which references the universal name-space of the SOAP envelope standard.</p>
<blockquote>
<p class="code">&lt;SOAP:Envelope xmlns:SOAP=&quot;urn:schemas-xmlsoap-org:soap.v1&quot;&gt;<br>
&nbsp;&nbsp;&nbsp; &lt;SOAP:Header&gt;&lt;/SOAP:Header&gt;<br>
&nbsp;&nbsp;&nbsp; &lt;SOAP:Body&gt;<br>
<font color="#808080">&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;GetSalesTax&gt;<br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;SalesTotal&gt;100&lt;/SalesTotal&gt;<br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;GetSalesTax&gt;</font><br>
&nbsp;&nbsp;&nbsp; &lt;/SOAP:Body&gt;<br>
&lt;/SOAP:Envelope&gt;</p>
</blockquote>
<p>Finally, to complete our SOAP request document, we will add a name-space
reference to our method call, which points to the object which contains the
method - the equivalent of the object declaration (Dim objTax As New CTaxCalc)
in our MTS method call.</p>
<blockquote>
<p class="code"><font color="#808080">&lt;SOAP:Envelope xmlns:SOAP=&quot;urn:schemas-xmlsoap-org:soap.v1&quot;&gt;<br>
&nbsp;&nbsp;&nbsp; &lt;SOAP:Header&gt;&lt;/SOAP:Header&gt;<br>
&nbsp;&nbsp;&nbsp; &lt;SOAP:Body&gt;</font><br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;m:GetSalesTax xmlns:m=&quot;urn:myserver/soap:TaxCalc&quot;&gt;<br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;SalesTotal&gt;100&lt;/SalesTotal&gt;<br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;/m:GetSalesTax&gt;<br>
<font color="#808080">&nbsp;&nbsp;&nbsp; &lt;/SOAP:Body&gt;<br>
&lt;/SOAP:Envelope&gt;</font></p>
</blockquote>
<p>Now that we have built our SOAP request document, we are ready to send it to
the server. The request is a simple HTTP post - just like posting a web form in
your internet browser. Of course, your internet browser masks all the complexity
of sending a form to a server; and in the longer-term .NET will mask this
process from your VB code too. But for now, we need to do the job ourselves; and
I have been using Microsoft's XML HTTP Request object to give me a helping hand.
(The XMLHTTPRequest is an object within the MSXML class library (MSXML.DLL), and
it comes with IE5.) Assuming that <i>strEnvelope</i> contains the XML document
described above, the request is formatted thus:</p>
<blockquote>
  <p class="code"><font color="#000080">Dim</font> objHTTP <font color="#000080">As
  New</font> MSXML.XMLHTTPRequest<br>
  <font color="#000080">Dim</font> strEnvelope <font color="#000080">As String</font><br>
  <br>
  <font color="#008000">'Set up to post to our localhost server</font><br>
  objHTTP.open &quot;post&quot;, &quot;http://localhost/soap/soap.asp&quot;<br>
  <br>
  <font color="#008000">'Set a standard SOAP/ XML header for the content-type</font><br>
  objHTTP.setRequestHeader &quot;Content-Type&quot;, &quot;text/xml&quot;<br>
  <br>
  <font color="#008000">'Set a header for the method to be called</font><br>
  objHTTP.setRequestHeader &quot;SOAPMethodName&quot;, _<br>
  &quot;urn:myserver/soap:TaxCalc#GetSalesTax&quot;<br>
  <br>
  <font color="#008000">'Make the SOAP call</font><br>
  objHTTP.send strEnvelope<br>
  <br>
  <font color="#008000">'Get the return value</font><br>
  strReturn = objHTTP.responseBody</p>
</blockquote>
<p>Now that we have sent our request,
we move to the server, to see how we set up a SOAP service to listen and respond
to our calls.</p>
<p class="ChapterH2">The Server</p>
<p>Our server needs to be configured to accept the HTTP post sent by the client.
It will be noted that we direct the client's post to a URL on our local server -
http://localhost/soap.asp. So our first job is to create this page, to listen
for, and process, SOAP calls to our server. We know that our ASP will receive an
XML document, in the form of an HTTP post, with a method name (GetSalesTax), and
a parameter (SalesTotal). So, to write our basic listener service, all we need
do is deconstruct the body of the request (the SOAP envelope) and pull out the
value of the SalesTotal parameter.</p>
<p>The SOAP envelope posted by the client is contained in the body of the
request, or in ASP terms the Request object; and because it is XML, we can load
it into an instance of Microsoft's XMLDOM in our ASP. Soap.asp begins like this:</p>
<blockquote>
<p class="code"><font color="#000080">Set</font> objReq =
Server.CreateObject(&quot;Microsoft.XMLDOM&quot;)<br>
objReq.Load Request</p>
</blockquote>
<p>Thus objReq contains the SOAP envelope we created on the client, and we can
extract the value of the SalesTotal parameter by running an XSL pattern query,
using the SelectSingleNode method of the XML DOM object :</p>
<blockquote>
<p class="code">strQuery = &quot;SOAP:Envelope/SOAP:Body/m:GetSalesTax/SalesTotal&quot;&nbsp;<br>
varSalesTotal = objReq.SelectSingleNode(strQuery).Text</p>
</blockquote>
<p>With the parameter extracted, we
can make our calculation to get the sales-tax:</p>
<blockquote>
<p class="code">varSalesTax = varSalesTotal * 0.04</p>
</blockquote>
<p>Now we have the return value for sales-tax - the <i>response</i> - ready to
pass back to the client, but as with the request, we need to format this
response correctly, in order to comply with the SOAP standard. The SOAP response
envelope conforms to a format-type identical to the request. The only difference
is that the &quot;IN&quot; parameter (SalesTotal, in our case) is replaced by an
&quot;OUT&quot; parameter - SalesTax, and the method name indicates that the
document is a <i>response</i>:</p>
<blockquote>
<p class="code"><font color="#808080">&lt;SOAP:Envelope xmlns:SOAP=&quot;urn:schemas-xmlsoap-org:soap.v1&quot;&gt;<br>
&nbsp;&nbsp;&nbsp; &lt;SOAP:Header&gt;&lt;/SOAP:Header&gt;<br>
&nbsp;&nbsp;&nbsp; &lt;SOAP:Body&gt;</font><br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;m:GetSalesTaxResponse xmlns:m=&quot;urn:myserver/soap:TaxCalc&quot;&gt;<br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;SalesTax&gt;4&lt;/SalesTax&gt;<br>
&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;/m:GetSalesTaxResponse&gt;<br>
<font color="#808080">&nbsp;&nbsp;&nbsp; &lt;/SOAP:Body&gt;<br>
&lt;/SOAP:Envelope&gt;</font></p>
</blockquote>
<p>We can build up this document, either by string-concatenation, or by creating
a new instance of a DOM, and appending the appropriate nodes.</p>
<p>Back on the client, the response is received, and can be decoded by
extracting the appropriate node from the Envelope document:</p>
<blockquote>
  <p class="code"><font color="#000080">Dim</font> objReturn <font color="#000080">As
  New</font> MSXML.DomDocument<br>
  objReturn.LoadXML strReturn</p>
  <p class="code">strQuery = _<br>
  &quot;SOAP:Envelope/SOAP:Body/m:GetSalesTaxResponse/SalesTax&quot;<br>
  dblTax = objReturn.SelectSingleNode(strQuery).Text</p>
</blockquote>
<p>And that's it! A functional, compliant SOAP service in a few easy steps. Of
course, the service that we have provided is far from sophisticated, but that is
to miss the point. In the not-too-distant future, Visual Studio 7 will
completely mask the implementation of SOAP; but I believe that there is value in
understanding the way the engine turns beneath the hood. SOAP itself is a very
simple protocol; and I hope you leave this article with a better understanding
the infrastructure that lies behind SOAP, and the methodology through which
SOAP-based component services are going to be provided in the future.</p>
<p class="ChapterH2">Appendix 1: VB Client Code</p>
<p>NB - This project contains a single module, with a Sub Main start-up object,
and references the MSXML library.&nbsp;</p>
<p align="center"><span style="font-size:10.0pt;mso-bidi-font-size:12.0pt;
font-family:Verdana;mso-fareast-font-family:&quot;Times New Roman&quot;;mso-bidi-font-family:
&quot;Times New Roman&quot;;mso-ansi-language:EN-US;mso-fareast-language:EN-US;
mso-bidi-language:AR-SA">
<img width="449" height="357" src="images/soap_s1.jpg" v:shapes="_x0000_i1025"></span></p>
<p>The code assumes you are running a <i>local</i> web server (<i>localhost</i>),
and that <i>soap.asp</i> is located in the web root, as described below.</p>
<div align="center">
  <center>
  <table border="0" width="90%" bgcolor="#EFEFEF" cellspacing="0" cellpadding="4">
    <tr>
      <td width="100%">
        <blockquote>
          <p class="code"><font color="#000080">Sub</font> Main()<br>
          <br>
          &nbsp;&nbsp; <font color="#000080">Dim</font> objHTTP <font color="#000080">As
          New</font> MSXML.XMLHTTPRequest<br>
          &nbsp;&nbsp; <font color="#000080">Dim</font> strEnvelope <font color="#000080">As
          String<br>
          &nbsp;&nbsp; Dim</font> strReturn <font color="#000080">As String<br>
          &nbsp;&nbsp; Dim</font> objReturn <font color="#000080">As New</font>
          MSXML.DOMDocument<br>
          &nbsp;&nbsp; <font color="#000080">Dim</font> dblTax <font color="#000080">As
          Double<br>
          &nbsp;&nbsp; Dim</font> strQuery <font color="#000080">As String</font><br>
          <br>
          <font color="#008000">&nbsp;&nbsp; 'Create the SOAP Envelope</font><br>
          &nbsp;&nbsp; strEnvelope = _<br>
          &nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &quot;&lt;SOAP:Envelope xmlns:SOAP=&quot;&quot;urn:schemas-xmlsoap-org:soap.v1&quot;&quot;&gt;&quot;
          &amp; _<br>
          &nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &quot;&lt;SOAP:Header&gt;&lt;/SOAP:Header&gt;&quot;
          &amp; _<br>
