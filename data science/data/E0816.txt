      <h2><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">#Definition&nbsp;<font size="3"><br>
      Difficulty: 4 / 10</font><!--mstheme--></font></h2>
<p align="left"><i>What can and can't macros do? Not all compilers agree.</i></p>
<p align="center"><img border="0" src="../images/h-line.gif" width="248" height="2"></p>
<h3><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">Problem<!--mstheme--></font></h3>
<h4><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">JG Question<!--mstheme--></font></h4>
<p><b><font size="4">1.</font></b> Demonstrate how to write a simple <code>max()</code>
preprocessor macro that takes two arguments and evaluates to the one that is
greater, using normal <code>&lt;</code> comparison. What are the usual pitfalls in
writing such a macro?</p>
<h4><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">Guru Question<!--mstheme--></font></h4>
<p><b><font size="4">2.</font></b> What can a preprocessor macro not create? Why
not?</p>
<p align="center"><img border="0" src="../images/h-line.gif" width="248" height="2"></p>
<h3><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666"><a name="Solution"></a>Solution<!--mstheme--></font></h3>

<h4><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">Common Macro Pitfalls<!--mstheme--></font></h4>

<p><font color="#999933"><b><font size="4">1.</font></b> Demonstrate how to write a simple <code>max()</code>
preprocessor macro that takes two arguments and evaluates to the one that is
greater, using normal <code>&lt;</code> comparison. What are the usual pitfalls in
writing such a macro?</font></p>
<p>There are four major pitfalls, besides several further drawbacks. Focusing on
the pitfalls first, here are common ways to go wrong when writing a macro.</p>
<p><b>1. Don't forget to put parentheses around arguments.</b></p>
<!--mstheme--></font><pre>  // Example 1(a): Paren pitfall #1: arguments
  //
  #define max(a,b) a &lt; b ? b : a</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>The problem here is that the uses of the parameters a and b are not fully
parenthesized. Macros only do straight textual substitution, so this can cause
some unexpected results. For example:</p>
<!--mstheme--></font><pre>  max( i += 3, j )</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>expands to the following:</p>
<!--mstheme--></font><pre>  i += 3 &lt; j ? j : i += 3</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>which, because of operator precedence and language rules, actually means:</p>
<!--mstheme--></font><pre>  i += ((3 &lt; j) ? j : i += 3)</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>This could cause some long debugging sessions.</p>
<p><b>2. Don't forget to put parentheses around the whole expression.</b></p>
<p>Fixing the first problem, we still fall prey to another subtlety:</p>
<!--mstheme--></font><pre>  // Example 1(b): Paren pitfall #2: expansion
  //
  #define max(a,b) (a) &lt; (b) ? (b) : (a)</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>The problem now is that the entire expansion is not correctly parenthesized.
For example:</p>
<!--mstheme--></font><pre>  k = max( i, j ) + 42;</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>expands to the following:</p>
<!--mstheme--></font><pre>  k = (i) &lt; (j) ? (j) : (i) + 42;</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>which, because of operator precedence, actually means:</p>
<!--mstheme--></font><pre>  k = (((i) &lt; (j)) ? (j) : ((i) + 42));</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>If i &gt;= j, k is assigned the value of i+42, as intended. But if i &lt; j,
k is assigned the value of j.</p>
<p><b>3. Be aware of the results of any possible multiple evaluation.</b></p>
<p>We can fix problem #2 by putting parentheses around the entire macro
expansion, but this leaves us with yet another problem:</p>
<!--mstheme--></font><pre>  // Example 1(c): Multiple argument evaluation
  //
  #define max(a,b) ((a) &lt; (b) ? (b) : (a))</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>Now, consider what happens if one or both of the expressions has side
effects:</p>
<!--mstheme--></font><pre>  max( ++i, j )</pre><!--mstheme--><font face="Arial, Arial, Helvetica"><p>If the result of ++i &gt;= j, i gets incremented twice, which is
probably not what the programmer intended:</p>
<!--mstheme--></font><pre>  ((++i) &lt; (j) ? (j) : (++i))</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>Similarly, consider the code:</p>
<!--mstheme--></font><pre>  max( f(), pi )</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>which expands to:</p>
<!--mstheme--></font><pre>  ((f()) &lt; (pi) ? (pi) : (f()))</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>If the result of f() &gt;= pi, f() gets executed twice, which is almost
certainly inefficient and often actually wrong.</p>
<p>Alas, although we could work around the first two problems, this one is a
corker -- there is no solution as long as max is a macro.</p>
<p><b>4. Beware scope.</b></p>
<p>Finally, macros don't care about scope. (They don't care about much of
anything; see <a href="063.htm">GotW #63</a>.) They just perform textual
substitution no matter where the text may be. This means that, if we use macros
at all, we have to be very careful about what we name them. In particular, the
biggest problem with the max macro is that it is highly likely to interfere with
the standard max() function template:</p>
<!--mstheme--></font><pre>  // Example 1(d): Name tromping
  //
  #define max(a,b) ((a) &lt; (b) ? (b) : (a))

  #include &lt;algorithm&gt; // oops!</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>The problem is that, inside header &lt;algorithm&gt;, there will be something
like the following:</p>
<!--mstheme--></font><pre>  template&lt;class T&gt; const T&amp; 
  max(const T&amp; a, const T&amp; b);</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>which the macro &quot;helpfully&quot; turns into an uncompilable mess:</p>
<!--mstheme--></font><pre>  template&lt;class T&gt; const T&amp; 
  ((const T&amp; a) &lt; (const T&amp; b) ? (const T&amp; b) : (const T&amp; a));</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>If you think that's easy to avoid by putting your macro definition after all
#included header files (which really is a good idea in any case), just imagine
what the macro does to all your other code that happens to have variables or
other things that just happen to be named max.</p>
<p>If you have to write a macro, try to give it an unusual and hard-to-spell
name that will be less likely to tromp on other names.</p>
<h4><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">Other Macro Drawbacks<!--mstheme--></font></h4>

<p>There are a few other major things a macro can't do:</p>
<p><b>5. Macros can't recurse.</b></p>
<p>We can write a recursive function, but it's impossible to write a recursive
macro. As the C++ standard says, in 16.3.4/2:</p>
<blockquote>
  <p><i>If the name of the macro being replaced is found during this scan of the
  replacement list (not including the rest of the source file's pre- processing
  tokens), it is not replaced. Further, if any nested replacements encounter the
  name of the macro being replaced, it is not replaced. These nonreplaced macro
  name preprocessing tokens are no longer available for further replacement even
  if they are later (re)examined in contexts in which that macro name
  preprocessing token would otherwise have been replaced.</i></p>
</blockquote>
<p><b>6. Macros don't have addresses.</b></p>
<p>It's possible to form a pointer to any free or member function (for example,
to use it as a predicate), but it's not possible to form a pointer to a macro
because a macro has no address. Why not should be obvious: Macros aren't code. A
macro doesn't have any existence of its own, because all it is is a glorified
(and not particularly glorious) text substitution rule.</p>
<p><b>7. Macros are debugger-unfriendly.</b></p>
<p>Besides the fact that macros change the underlying code before the compiler
gets a chance to see it, and therefore can wreak havoc with variable and other
names, a macro can't be stepped into during debugging.</p>
<p>Have you heard the one about the scientists who started experimenting on
lawyers instead of on laboratory macros? It was because...</p>
<h4><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">There Are Some Things Even a Macro Just Won't Do<!--mstheme--></font></h4>
<p>There are valid reasons to use macros (see <a href="032.htm">GotW #32</a>),
but there are limits. This brings us to the final question:</p>

<p><font color="#999933"><b><font size="4">2.</font></b> What can a preprocessor macro not create? Why
not?</font></p>

<p>In the standard, clause 2.1 defines the phases of translation. Preprocessing
directives and macro expansions take place in phase 4. Thus, on a compliant
compiler, it is not possible for a macro to create:</p>

<blockquote>

<p>- a trigraph (trigraphs are replaced in phase 1);</p>

  <p>- a universal character name (\uXXX, replaced in phase 1);</p>

  <p>- an end-of-line line-splicing backslash (replaced in phase 2);</p>

<p>- a comment (replaced in phase 3);</p>

<p>- another macro; or</p>

<p>- changes to a character literal or string literal via macro names inside the
strings. For this last point, as noted in 16.3/8 footnote 7:</p>

</blockquote>
<blockquote>
  <blockquote>

<p><i>Since, by macro-replacement time, all character literals and string
literals are preprocessing tokens, not sequences possibly containing
identifier-like subsequences (see 2.1.1.2, translation phases), they are never
scanned for macro names or parameters.</i></p>

  </blockquote>
</blockquote>
<p>A recent CUJ article<b><sup><a href="#1">[1]</a></sup></b> claimed that it's
possible for a macro to create a comment:</p>
<!--mstheme--></font><pre>  #define COMMENT SLASH(/)
  #define SLASH(s) /##s</pre><!--mstheme--><font face="Arial, Arial, Helvetica">

compilers</i>. Why? Because those compilers don't implement the phases of
