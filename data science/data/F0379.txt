<p><font face="Arial, Helvetica" size="+1"><b>Dreams do come true - The MS Office Object Models: Part 2</b></font></p>
<p><i>By <a href="/php-bin/authorbio.php?authorid=48&associd=104&assoctype=2">Ronald Garlit</a></i></p>
<p><font face="Arial, Helvetica" size="+1"><b></b></font></p>

<p>In the kick off article I talked about several of the object models in the Microsoft Office 97.  I mentioned numerous counts of properties and methods of the objects in those assorted models.  I’ve decided to focus on the largest and least documented of the suite.  We are going to be discussing the Word Object Model.</p>

<p>But we have a new twist here.  You see it’s now the year 2000.  Everything that has a model or product name has a 2000 in it.  I’m Y2K’d out.  I didn’t work on any of those date fix projects because we did it right years ago.  We were designing libraries for DOS C compilers that took the year into account.</p>

<p>However, this ever evolving Word object model has it’s own numbers that scare us.  I mentioned before that the Word 8.0 model had 188 object and 3137 properties and methods.   I was going to be really cool and update these numbers for the Word 9.0 model.  But I give up.  You see since the release of Office 2000 Microsoft has release a developer’s version of the product with new and improve documentation.</p>

<p>Using the documentation with Office 2000 premium version I counted almost 200 objects in the Word 9.0 model.  But I now find is that my counts could be off.  You see Word 2000 has added “Click and Type” formatting plus who knows what else and slid it in without any real documentation (Just like the DOS days).  I couldn’t find any references to it in the object models documentation that I had.</p>

<p>But it is now there in MSDN and I’ll bet the NEW developer version is loaded with information.  Right now the only information is for paragraph properties but no methods.  I think this is one of those items we should ignore for now on the programming side because the tools are not there for us yet.  But when Microsoft has developed this feature enough, we could have a whole new way of positioning objects in a document.</p>

<p>For now we will be teased and left to imagine what we could do with this feature while we type our documents.  So let have some fun.  Let me tell you a little about the Word Application Object.  Word 8.0 it had 139 properties and methods plus two (it could be argued 3) events.  But in the Word 9.0 application object there are now by my count 159 properties and methods and twelve events.  So I think it’s safe to say that we have a vastly more improved MS Word on our hands.</p>

<p>So here is what I’ve done.  I created a project designed to fulfill the promise of showing how to replace the VB printer object with MS Word.  I’ve also created the debug print  statements in the application object’s events for Word.  You can bring Word up again by running the program after you first comment out the line Objword.Quit in the Document_New( ) procedure.  Then you can manipulate the visible version of MS Word and watch the events in the immediate window of Visual Basic.  Which should display the following:</p>

<p>Creating a document with NORMAL.DOT<br>
Word windowActivate event fired<br>
Word NewDocument Event fired<br>
running SQL Statement: SELECT Name, Price FROM Product<br>
Word Document Change event fired<br>
converting data to table<br>
Word DocumentBeforePrint event fired<br>
 1 <br>
 0 <br>
Word DocumentBeforeclose event fired<br>
Word WindowDeactivate event fired<br>
Word Document Change event fired<br>
Word Quiting Event Fired</p>

<p>If your not using Word 2000 you can comment out all the events that start as “Objword_” except the DocumentChange and Quit events and the program will work with Word 97.</>

<p>You can the find code in the demo at the end. Below is the an explaination of the code in the Sub Document_New( ) you will find it in the Form1 General Declarations section.  This is the heart of printing with MS Word.</p>

<p>First we create a instance of Word:</p>

</font><pre>
    <font color="#007F00">' Create a new instance of the Word</font>
    <font color="#007F00">' Note:  I am NOT using <font color="#00007F">GetObject</font> or <font color="#00007F">CreateObject</font></font>
    <font color="#007F00">' on purpose.</font>
    <font color="#00007F">Set</font> Objword = <font color="#00007F">New</font> Word.Application
</pre><FONT SIZE="-1" FACE="verdana, arial, helvetica">
<p>Then we set our window state for the instance.  We make it visible here for testing purposes.  But it is should be set to be invisible in a finished application:</p>
</font><pre>
    <font color="#00007F">With</font> Objword
        <font color="#007F00">' No explaination needed</font>
        .WindowState = wdWindowStateNormal
        .Resize 300, 300
        .Move 100, 100
        <font color="#007F00">' toggle visible property of the object</font>
        <font color="#007F00">' True during development and testing</font>
        <font color="#007F00">' False when deployed because we don't want it seen</font>
        <font color="#007F00">' also some speed reason too.</font>
        .Visible = True</pre><FONT SIZE="-1" FACE="verdana, arial, helvetica">

<p>Next we create a Document to put our report in:</p>

</font><pre>.Documents.<font color="#00007F">Add</font> "normal.dot", , , True</pre><FONT SIZE="-1" FACE="verdana, arial, helvetica">

<p>Now we are ready to open a database and get a record set:</p>
</font><pre>
        sDBPath = App.Path & "\DemoData.mdb"
        <font color="#007F00">' <font color="#00007F">Set</font> up the ADO connection information</font>
        sConnection = "Provider=Microsoft.Jet.OLEDB.4.0;" & _
            "Persist Security Info=False;" & _
            "Data Source=" & sDBPath
        <font color="#007F00">' Make the ADO connection</font>
        Conn.<font color="#00007F">Open</font> sConnection
        <font color="#007F00">' <font color="#00007F">Set</font> up the SQL Statement</font>
        SQL = "SELECT <font color="#00007F">Name</font>, Price FROM Product"
        <font color="#007F00">' Make a recordset based on the SQL statement</font>
        rsProducts.<font color="#00007F">Open</font> SQL, Conn, adOpenStatic, adLockReadOnly</pre><FONT SIZE="-1" FACE="verdana, arial, helvetica">

<p>Next we write that data in a delimited form to the ActiveDocument:</p>
</font><pre>
        <font color="#00007F">Do</font> <font color="#00007F">While</font> <font color="#00007F">Not</font> rsProducts.<font color="#00007F">EOF</font>
            <font color="#007F00">' The simpler slower one line of text at a time</font>
            <font color="#007F00">' notice it is delimited using vbTab constant</font>
            <font color="#007F00">' This is faster but there is a faster method.</font>
            sLine = rsProducts("Name") & vbTab & _
                <font color="#00007F">Format</font>(rsProducts("Price"), "Currency") & vbCrLf
            <font color="#007F00">' Write a line of TAB delimited text</font>
            .ActiveDocument.Range.InsertAfter sLine
            <font color="#007F00">' Move to the next record in the recordset</font>
            rsProducts.MoveNext
        <font color="#00007F">Loop</font>    <font color="#007F00">'}-> <font color="#00007F">Do</font> <font color="#00007F">While</font> <font color="#00007F">Not</font> rsProducts.<font color="#00007F">EOF</font></font>
</pre><FONT SIZE="-1" FACE="verdana, arial, helvetica">
<p>Now we do some Word Magic.  Call the Range Objects ConvertToTable Method.</p>
</font><pre>
        .ActiveDocument.Range.ConvertToTable</pre><FONT SIZE="-1" FACE="verdana, arial, helvetica">

<p>Then we print out the report to the default printer:</p>

</font><pre>        .ActiveDocument.PrintOut
    <font color="#00007F">End</font> <font color="#00007F">With</font>  <font color="#007F00">' word</font></pre><FONT SIZE="-1" FACE="verdana, arial, helvetica">

<p>And we Quit word.</p>
</font><pre>
    <font color="#007F00">' This loop performs testing of printing status</font>
    <font color="#007F00">' without this testing to delay the use of the Quit method</font>
    <font color="#007F00">' you would get an error that would cause you heartache</font>
    <font color="#007F00">' if word was invisible!!!!!!</font>
    <font color="#00007F">Do</font> <font color="#00007F">While</font> Objword.BackgroundPrintingStatus > 0
        <font color="#00007F">Debug</font>.<font color="#00007F">Print</font> Objword.BackgroundPrintingStatus
    <font color="#00007F">Loop</font>    <font color="#007F00">'}-> <font color="#00007F">Do</font> <font color="#00007F">While</font> Objword.BackgroundPrintingStatus > 0</font>
    Objword.Quit wdDoNotSaveChanges</pre><FONT SIZE="-1" FACE="verdana, arial, helvetica">

<p>That beats the heck out of using the printer object.  As you can see while running the program that it is rather quick once your done accessing the database.  All you are really waiting on is for Word to background print to the spooler and word closes.  Leaving you to do other things while the printer outputs the report.</p>

<p>So now I’ve given you a test program to study the Word application events and one of the quickest ways to build a report from a database.  By studying this program you learn how to write text to the Word document in TAB delimited form.  Which happens to be the default for the ConvertToTable method.  You show really look the help over on this method as it has many options.  It will amaze you.</p>

<p>So there you have it.  I asked for it, and got it.  More tools than I could ever have asked for.  Let alone have time to write about.  If you were counting, we only used about nine properties and methods from the Word 9.0 Object Model out of well over 3000 (bet you thought I was going to say 2000 right?).</p>

<p>So instead of surfing with an Internet Browser tonight.  May I suggest surfing you windows system with the Object Browser?</p>
