      <h2><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">Using Standard Containers&nbsp;<font size="3"><br>
      Difficulty: 6 / 10</font><!--mstheme--></font></h2>
<p><i>Oil and water just don't mix. Do pointers and standard containers mix any
better?</i></p>
<p align="center"><img border="0" src="../images/h-line.gif" width="248" height="2"></p>

<h3><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">Problem<!--mstheme--></font></h3>

<h4><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">JG Questions<!--mstheme--></font></h4>

<p>Consider the following code:</p>
<!--mstheme--></font><pre>    void f( vector&lt;char&gt;&amp; v ) {
      char* pc = &amp;v[0];
      // ... later, uses pc ...
    }</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p><b><font size="4">1.</font></b>   Is this code valid?</p>
<p><b><font size="4">2.</font></b>   Whether it's valid or not, how could it be
improved?</p>
<h4><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">Guru Question<!--mstheme--></font></h4>
<p><b><font size="4">3.</font></b>   Consider the following code:</p>
<!--mstheme--></font><pre>    template&lt;class T = std::vector&lt;T&gt; &gt;
    void f( T&amp; t ) {
      typename T::value_type* p1 = &amp;t[0];
      typename T::value_type* p2 = &amp;*t.begin();
      // ... later, uses p1 and p2 ...
    }</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>Is this code valid? Discuss.</p>
<p align="center"><img border="0" src="../images/h-line.gif" width="248" height="2"></p>
<h3><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666"><a name="Solution"></a>Solution<!--mstheme--></font></h3>

<p><font color="#999933">Consider the following code:</font></p>
<!--mstheme--></font><pre><font color="#999933">    void f( vector&lt;char&gt;&amp; v ) {
      char* pc = &amp;v[0];
      // ... later, uses pc ...
    }</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p><font color="#999933"><b><font size="4">1.</font></b>   Is this code valid?</font></p>
<p>Yes, as long as v is nonempty.</p>
<p>Why is this code legal? The reason comes right out of the standard's
Container and Sequence requirements: If operator[] is provided for a given kind
of sequence (which it is in the case of std::vector&lt;char&gt;), it must return
a &quot;reference&quot;. In turn, the &quot;reference&quot; must be an lvalue of
type T (here char), which can have its address taken.</p>
<h4><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">Why Take Pointers or References Into Containers?<!--mstheme--></font></h4>
<p>Many programmers are surprised by this kind of code the first time they see
it, but there's nothing wrong with this technique in itself. As long as you
remain aware of when the pointers might be invalidated, which is pretty much
whenever iterators would be invalidated, this technique is not evil, it is not
immoral, and it's not even fattening. On the contrary, it can be quite useful.</p>
<p>There are cases where it can be legal (and, more to the point, where it makes
perfect sense) to have pointers or references into containers. A common case is
when you read a data structure into memory on program startup and then never
modify it, but you need to access it frequently in different ways. In such cases
it can make sense to have additional data structures that contain pointers into
the main container to optimize different access methods. I'll give an example as
part of the discussion of Question #2.</p>
<p><font color="#999933"><b><font size="4">2.</font></b>   Whether it's valid or
not, how could it be improved?</font></p>
<p>In general, it's not a bad guideline to prefer to use iterators instead of
pointers when you want to point at an object that's inside a container. After
all, iterators are invalidated at mostly the same times and the same ways as
pointers.</p>
<p>There are two main potential drawbacks to this method. If either applies in
your case, continue to use pointers.</p>
<p>1. You can't always conveniently use an iterator where you can use a pointer.
(See example below.)</p>
<p>2. Using iterators might incur extra space and performance overhead, in cases
where the iterator is an object and not just a bald pointer.</p>
<p>Example: Say that you have a map&lt;Name,PhoneNumber&gt; that, given a name,
makes it easy to look up a phone number. What if you need to do the reverse
lookup too? One solution is to build a second structure, perhaps a
map&lt;PhoneNumber*,Name*,DerefFunctor&gt; that enables the reverse lookup but
avoids doubling the storage overhead (no need to store each name and phone
number twice; the second structure simply has pointers into the first).</p>
<p>Note that, in the above example, it would be difficult to use iterators
instead of pointers. Why? Because the natural candidate,
map&lt;Name,PhoneNumber&gt;::iterator, points to a pair&lt;Name,PhoneNumber&gt;,
and there's no handy way to get an iterator to just the name or phone number
part individually.</p>
<p>This brings us to the next (and in my opinion most interesting) point of
today's lesson:</p>
<h4><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">When Is a Container Not a Container?<!--mstheme--></font></h4>
<p><font color="#999933"><b><font size="4">3.</font></b>   Consider the
following code:</font></p>
<!--mstheme--></font><pre><font color="#999933">    template&lt;class T&gt;</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>The above line has two problems:</p>
<p>a) The easy one is that it doesn't make sense to define T in terms of itself.
That red herring may have successfully tricked you into missing the more
fundamental problem, namely:</p>
<p>b) Function templates can't have default arguments.</p>
<p>Now consider the rest of the function as if it were introduced with
&quot;template&lt;class T&gt;&quot; only:</p>
<!--mstheme--></font><pre><font color="#999933">    void f( T&amp; t ) {
      typename T::value_type* p1 = &amp;t[0];
      typename T::value_type* p2 = &amp;*t.begin();
      // ... later, uses p1 and p2 ...
    }</font></pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p><font color="#999933">Is this code valid? Discuss.</font></p>
<p>The short answer is: Sometimes.</p>
<p>One instructive way to look at this problem is to think about what kinds of T
and t make the code valid. At compile time, what characteristics and abilities
must T have? At runtime, what characteristics must t have? Let's do some
detective work.</p>
<p>At compile time:</p>
<p>a) To make the expression &amp;t[0] valid, T::operator[] must exist and must
return something that understands operator&amp;.</p>
<p>In particular, this is true of containers that meet the standard's Container
requirements and implement the optional operator[], because that operator must
return a reference to the contained object. By definition, you can then take the
contained object's address.</p>
<p>b) To make the expression &amp;*t.begin() valid, T::begin() must exist, and
it must return something that understands operator*, which in turn must return
something that understands operator&amp;.</p>
<p>In particular, this is true of containers that meet the standard's iterator
requirements, because the iterator returned by begin() must, when dereferenced
using operator*, return a reference to the contained object. By definition, you
can then take the contained object's address.</p>
<p>Further, at runtime:</p>
<p>c) To make the expression &amp;t[0] safe, the t object must be in a suitable
state for calling t[0]. In particular, if T is something like
std::vector&lt;int&gt;, then the vector must not be empty.</p>
<p>d) To make the expression &amp;*t.begin() safe, the t object must be in a
suitable state for calling t.begin() and dereferencing the result. In
particular, if T is something like std::vector&lt;int&gt;, then the vector must
be nonempty.</p>
<h4><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">Which Brings Us to the Embarrassing Part<!--mstheme--></font></h4>
<p>The code in Question #3 will work for every container in the standard library
that supports operator[] -- and, if you take away the line containing
&quot;&amp;t[0]&quot;, it will work for every container in the standard library,
bar none -- EXCEPT for std::vector&lt;bool&gt;. In fact, the following template:</p>
<!--mstheme--></font><pre>    template&lt;class T&gt;
    void g( vector&lt;T&gt;&amp; v ) {
      T* p = &amp;*t.begin();
      // ... do something with p ...
    }</pre><!--mstheme--><font face="Arial, Arial, Helvetica">
<p>works for every type EXCEPT bool. If this seems a little strange to you,
you're not alone.</p>
<h4><!--mstheme--><font face="Verdana, Arial, Helvetica" color="#006666">What About bool?<!--mstheme--></font></h4>
<p>Unfortunately, there's something a little embarrassing about the C++ standard
library: not all of the templates it defines that look like containers actually
are Containers. In particular,</p>
<blockquote>
  <p align="center"><b>std::vector&lt;bool&gt; IS NOT a Container</b></p>
</blockquote>
<p>because it does not meet the standard library's requirements for Containers.
It appears in the Containers and Sequences section of the standard without any
note to indicate that it is neither.</p>
<p>(If anyone else had written vector&lt;bool&gt;, it would have been called
&quot;nonconforming&quot; and &quot;nonstandard.&quot; Well, it's in the
standard, so that makes it a little bit harder to call it those names at this
point, but some of us try anyway in the hopes that it will eventually get
cleaned up. The correct solution is to remove the vector&lt;bool&gt;
specialization requirement so that vector&lt;bool&gt; really is a vector of
plain old bools. Besides, it's mostly redundant: std::bitset was designed for
this kind of thing.)</p>
<p>The reason std::vector&lt;bool&gt; is nonconforming is that it pulls tricks
under the covers in an attempt to optimize for space: Instead of storing a full
char or int for every bool<b><sup><a href="#1">[1]</a></sup></b> (taking up at
least 8 times the space, on platforms with 8-bit chars), it packs the bools and
stores them as individual bits (inside, say, chars) in its internal
representation. One consequence of this is that it can't just return a normal
bool&amp; from its operator[] or its dereferenced iterators<b><a href="#2"><sup>[2]</sup></a></b>;
instead, it has to play games with a helper &quot;proxy&quot; class that is
bool-like but is definitely not a bool. Unfortunately, that also means that
access into a vector&lt;bool&gt; is slower, because we have to deal with proxies
instead of direct pointers and references.</p>
<p>All of the above trickery has the following disadvantages:</p>
<p>1. std::vector&lt;bool&gt; is not a Container. 'Nuff said.</p>
<p>2. std::vector&lt;bool&gt; attempts to illustrate how to write proxied
containers that pull tricks under the covers. Unfortunately, that's not a sound
idea, because by definition that violates the Container requirements. (See #1.)
(The Container requirements were never changed to enable proxied containers to
actually be conforming, and as far as I know that decision was deliberate.)</p>
<p>3. std::vector&lt;bool&gt;'s name is misleading because the things inside
aren't even standard bools. A standard bool is at least as big as a char, so
that it can be used &quot;normally.&quot; So, in fact, std::vector&lt;bool&gt;
does not even store bools, despite the name.</p>
<p>4. std::vector&lt;bool&gt; forces a specific optimization on all users by
enshrining it in the standard. That's not a good idea; different users have
different requirements, and now all users of vector&lt;bool&gt; must pay the
performance penalty even if they don't want or need the space savings.</p>
<p>Bottom line: If you care more about speed than you do about size, you
shouldn't use std::vector&lt;bool&gt;. Instead, you should hack around this
optimization by using a std::vector&lt;char&gt; or the like instead, which is
unfortunate but still the best you can do.</p>

<p><a name="2"></a>2. That's because there is not now a standard way to express
a pointer or a reference to a bit.</p>
