                            <h1 align="CENTER"> Java security evolution and concepts, Part 5</h1>
                            <h3 align="CENTER"> J2SE 1.4 offers numerous improvements to Java security</h3>
                        </p>

                        <BLOCKQUOTE>
<STRONG>Summary</STRONG><BR>


In Parts 1 through 4 of this series on Java security, Raghavan N. Srinivas discussed network and Java security concepts; Part 3 took a detailed look at applet security and optional packages. In this article, Raghavan introduces J2SE (Java 2 Platform, Standard Edition) 1.4's new security packages for certificate chain manipulation, along with  Generic Security Services, which includes a single sign-on framework over the network. (<em>3,200 words;</em> <strong>December 21, 2001</strong>)

                        <p><P>
<img align="left" width="23" height="29" src="/javaworld/abcs/E.gif" alt="E">arly on, Java security focused on resisting executable content threats -- security risks caused by malicious or poorly designed code. From that beginning, Java security measures evolved to provide security based on where code came from and who signed it. More recently, Java security has improved further to provide user and pluggable authentication, as well as SSL (Secure Socket Layer) and other key management and cryptography support. <P>
Java security evolution and concepts: Read the whole series! <P>
<ul> <li><a href="/javaworld/jw-04-2000/jw-0428-security.html">Part 1: Learn computer security concepts and terms in this introductory overview</a></li> <li><a href="/javaworld/jw-07-2000/jw-0728-security.html">Part 2: Discover the ins and outs of Java security</a></li> <li><a href="/javaworld/jw-12-2000/jw-1215-security.html">Part 3: Tackle Java applet security with confidence</a></li> <li><a href="/javaworld/jw-05-2001/jw-0525-security.html">Part 4: Learn how optional packages extend and enhance Java security</a></li> <li><a href="/javaworld/jw-12-2001/jw-1221-jdk4security.html">Part 5: J2SE 1.4 offers numerous improvements to Java security</a></li> </ul> <P>
This article, the last of five in the Java security series, follows the examples introduced in Part 3 and briefly revisits the optional packages covered in Part 4. By revisiting the examples from earlier articles, we can reinforce concepts and understand what has changed in the interim. <P>
After an overview of those security features that have changed between J2SE (Java 2 Platform, Standard Edition) 1.3 and the upcoming J2SE 1.4, this article digs deeper into the new packages included in 1.4. The first of the newer packages is the Java Certification Path (CertPath) API, which supports certificate chain manipulation. Second is the Java GSS-API (JGSS), which contains the Java bindings for the Generic Security Services Application Program Interface (GSS-API) for standardized access to a variety of security services and a framework for a single sign-on. <P><FONT SIZE="+1"><STRONG> Overview of changes </STRONG></FONT><BR>
First, let's quickly look at the changes between J2SE 1.3 and 1.4. <P><STRONG> Optional packages move into the 1.4 core </STRONG><BR>
As the most obvious change between J2SE 1.3 security and 1.4, the optional packages (examined in Part 4) will move into 1.4's core security structure: <P>
<ul> <li><strong>Java Authentication and Authorization Service (JAAS):</strong> A framework for performing user-based authentication</li> <li><strong>Java Cryptography Extension (JCE):</strong> A framework for using strong ciphers on a global basis</li> <li><strong>Java Secure Socket Extension (JSSE):</strong> Support for SSL and TLS (Transport Layer Security)</li> </ul> <P>
Each of the aforementioned packages will be available on every J2SE 1.4  installation, obviating the need to install them explicitly.
<P><STRONG> New 1.4 packages </STRONG><BR>
Besides moving the optional packages into the J2SE core, J2SE 1.4 will also feature two new security packages: <P>
<ul> <li><strong>Java CertPath API:</strong> Classes, methods, and interfaces to build and validate certificate chains</li> <li><strong>JGSS API:</strong> Java bindings for the Generic Security Services API</li> </ul> <P>
The bulk of this article will be devoted to looking into these two packages. <P><STRONG> Other changes </STRONG><BR>
As a result of JAAS becoming an integral part of core security, access control is no longer solely based on code source (i.e., where the code is from and who signed it), but also on who is running the code. As a result, both the Policy files and the graphical Policy tool have been enhanced to accommodate the changes. <P>
Next, strong cipher support is available via JCE and JSSE, albeit with some important control restrictions. In JCE, the cryptographic strengths are limited through the jurisdiction files by default. JSSE, for its part, now restricts the replacement of the default <code>SSLSocketFactory</code> and <code>SSLServerSocketFactory</code> classes. <P>
Moreover, you'll find support for dynamic policies. Prior to J2SE 1.4, the security policy was associated at the time of class loading. In 1.4, the security policy association is deferred until the security check becomes necessary. <P>
In addition, J2SE 1.4 includes some changes to the functionality of the security classes. As an example, the <code>javax.security.cert</code> class is available only for backward compatibility, with its functionality carried forward by the <code>java.security.cert</code> class. Newer applications should use this class. <P>
Finally, applet security as enforced by 1.4 is different, which we'll look into in the next section. <P><FONT SIZE="+1"><STRONG> Java Plug-in enhancements </STRONG></FONT><BR>
You'll see some changes in security between the J2SE 1.3 Java Plug-in and the 1.4 Java Plug-in (for more, see <a href="#sidebar1" name="return1">Sidebar 1: Java Plug-in Primer</a>). Since we are interested in security, I explore the changes by revisiting the signed applets developed in <a href="/javaworld/jw-12-2000/jw-1215-security.html">Part 3</a>. <P><STRONG> Security in the J2SE 1.3 Java Plug-in </STRONG><BR>
To illustrate how the J2SE 1.3 Plug-in handles applet security, let's first look at the <a href="http://www.javaworld.com/javaworld/jw-12-2000/security/writeFile.html" target="_blank"><code>writeFile</code></a> applet from Part 3 of this series. Execute the <code>writeFile</code> signed applet under the J2SE 1.3 Java Plug-in. Make sure that the policy files do not allow the <em>temp</em> file (<code>/tmp/foo</code> on Unix systems and <code>c:\tmpfoo</code> on Windows) to be modified. If the clock on your system is set correctly, the signed applet's signature should not validate because it has expired; therefore, the applet will not initialize. If you set your system's clock back to a date between December 13, 2000 and December 21, 2001 (that is, to a time when the certificate is valid), the signature validates, and you should see a dialog box asking whether the applet should run or not. <P>
As another example, run the signed applet discussed in "<a href="http://java.sun.com/security/signExample12/" target="_blank">JDK 1.2 -- Signed Applet Example</a>" under the J2SE 1.3 Java Plug-in. The applet has been signed by a key -- <em>Duke</em> -- whose public key has been self-certified. That means the Certification Authority (CA) used to sign the certificate is not trusted by default in the environment. As a result, the signature validation fails again, and the dialog box again fails to pop up. <P><STRONG> Security in the J2SE 1.4 Java Plug-in </STRONG><BR>
Now, let's run the same applet examples under the J2SE 1.4 Java Plug-in. <P>
When executing the example from Part 3, the signed applet with the expired certificate, the user sees a dialog box clearly indicating that the certificate is expired, as shown in Figure 1. Here, the ultimate decision to execute the applet is left to the user's discretion. <P>

The dialog box clearly warns against potential security holes but leaves the final decision of initializing the applet to the user. With the J2SE 1.3 Java Plug-in, the applet fails to initialize unless the user does something explicitly, such as setting the policies or adjusting the system clock.
<P>
Clearly, the J2SE 1.4 Java Plug-in offers the user greater flexibility. Moreover, in an intranet environment, you may no longer be required to sign applets with an external CA; a company-wide CA would suffice. The downside of this flexibility is that some responsibility for validating the signature now shifts to the user. That shift is probably all right, since technology can only so go far to validate a signature. The aspect of trusting a user based on the signature is an interpersonal issue and is left to the user anyway. Technology cannot address this important issue (and I hope it never will).
<P>
Now that we have covered some familiar ground, let's move on to J2SE 1.4's new packages. In the examples we looked at above, the Java Plug-in had to validate a certificate or a certificate chain. If, however, we had to build or validate a certificate chain in a program, we would use the Java CertPath API. <P>

 <font face="verdana, arial, helvetica" size="2"><P><P><H2 ALIGN="CENTER"> Sidebar 1: Java Plug-in primer </H2><P><P><P>A plug-in program adds functionality to a Web browser (or any other program). The Java Plug-in enables Java applets to run using the Java 2 Runtime Environment, Standard Edition (JRE), rather than the browser's default runtime. The Java Plug-in delivers several capabilities on different versions of Netscape Navigator and Internet Explorer, using the respective browsers' plug-in architecture and extension mechanisms. The network authentication protocol Kerberos -- named after a three-headed dog from Greek mythology -- answers some of these problems. <P>Kerberos provides mutual client-server authentication across an insecure connection, as well as privacy and data integrity. Services can be written to take advantage of Kerberos's secure features. <P>MIT, responsible for the development of Kerberos, offers several free versions of the protocol. You'll find it also available in source code form, as well as in several commercial implementations. <P>
